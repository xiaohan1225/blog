# 文章生产总控

使用这个技能，把一篇文章从选题推进到可发布状态。它不替代具体技能，而是负责编排：

- `technical-blog-writing`：负责技术文章结构、初稿、代码示例、图表设计。
- `huashu-proofreading`：负责三遍审校，降低 AI 味，优化逻辑和手机阅读节奏。
- `markdown-mermaid-to-images`：负责把 Markdown 里的 Mermaid 代码块转成 PNG 图片。
- `wechat-cover-generation`：负责生成微信公众号封面图或封面提示词。

## 总原则

- 先写内容，再审校，再做发布适配，最后做封面。
- Mermaid 图表在文章定稿前保留源码，不要过早转图片。
- 封面依赖最终标题和定位，放到最后处理。
- 每个阶段都要产出明确文件或结果，避免只停留在建议。

## 输入信息

开始前尽量收集：

- 文章主题：
- 目标读者：
- 文章类型：教程、解释型文章、对比评测、架构设计、复盘、观点文。
- 发布平台：公众号、博客、掘金、知乎等。
- 预期字数：
- 是否需要 Mermaid 图表：
- 是否需要代码示例：
- 是否需要公众号封面：
- 输出目录：

如果用户没有提供，默认：

- 文章类型：解释型文章。
- 发布平台：微信公众号。
- 预期字数：2000 到 3000 字。
- 需要 Mermaid 图表：按内容需要决定。
- 需要公众号封面：需要。

## 阶段 1：选题和大纲

调用 `technical-blog-writing`。

目标：

- 明确文章角度。
- 确定读者水平。
- 设计文章结构。
- 判断哪些地方需要图表、代码或案例。

输出：

```text
<文章名>-大纲.md
```

大纲至少包含：

- 标题候选 3 到 5 个。
- 目标读者。
- 核心观点。
- 章节结构。
- 需要配图或 Mermaid 图的位置。
- 需要查证的事实清单。

## 阶段 2：初稿写作

继续调用 `technical-blog-writing`。

目标：

- 按大纲写完整初稿。
- 技术内容要解释清楚，不堆概念。
- 需要图表时，先用 Mermaid 代码块表达。
- 如果有代码示例，确保代码块标注语言。

输出：

```text
<文章名>-初稿.md
```

写作要求：

- 开头直接进入问题，不写空泛背景。
- 每节都服务于核心观点。
- 图表用于解释流程、结构、决策或对比，不为装饰而画。
- 不确定的事实标记为待核查，不要编造。

## 阶段 3：事实和结构复查

调用 `huashu-proofreading` 的第一遍审校。

目标：

- 检查事实、时间、产品名、数据是否准确。
- 检查逻辑是否连贯。
- 检查章节是否重复、跑题或缺关键内容。

输出：

```text
<文章名>-结构审校.md
```

处理规则：

- 不确定信息必须明确标注或查证。
- 删除空泛段落。
- 调整章节顺序，让读者顺着问题自然读下去。

## 阶段 4：降 AI 味和风格审校

调用 `huashu-proofreading` 的第二遍和第三遍审校。

目标：

- 降低 AI 腔。
- 增加真人表达、经验感和具体细节。
- 优化手机阅读节奏。

输出：

```text
<文章名>-审校.md
```

审校重点：

- 删除“在当今时代”“值得注意的是”“综上所述”等套话。
- 长句拆短，段落控制在手机屏幕 3 到 5 行。
- 少用机械列表，多用自然叙述。
- 抽象表达改成具体例子、数字或场景。
- 保留技术准确性，不为了口语化牺牲事实。

## 阶段 5：Mermaid 转图片

只在文章内容基本定稿后调用 `markdown-mermaid-to-images`。

目标：

- 把 `<文章名>-审校.md` 里的 Mermaid 代码块转成 PNG。
- 替换原 Mermaid 代码块为图片引用。
- 保留 `.mmd` 源文件，方便后续改图。

输出：

```text
<文章名>-审校.md
<文章名>-审校-assets/mermaid/diagram-01.mmd
<文章名>-审校-assets/mermaid/diagram-01.png
```

验证：

```bash
rg -n '```mermaid' '<文章名>-审校.md' || true
ls -lh '<文章名>-审校-assets/mermaid'/*.png
```

要求：

- Markdown 里不再残留 Mermaid 代码块。
- PNG 数量和原 Mermaid 图数量一致。
- 图片路径使用相对路径，适合复制到 mdnice 或公众号编辑器。

## 阶段 6：公众号封面

调用 `wechat-cover-generation`。

目标：

- 根据最终标题、文章主题和受众生成封面图或封面提示词。
- 标题必须清晰，适合公众号列表小图预览。

输出：

```text
<文章名>-公众号封面.png
```

或者：

```text
<文章名>-封面提示词.md
```

封面信息从最终文章中提取：

- 主标题：优先使用最终标题。
- 副标题：使用关键词、模型名、栏目名或一句补充说明。
- 主视觉：从文章主题提炼，不要泛泛使用科技背景。
- 风格：和文章定位一致。

## 阶段 7：发布前检查

在交付前做一次总检查：

- 标题是否和正文一致。
- 开头是否足够直接。
- 事实和时间是否已核查。
- Mermaid 是否已转成图片。
- 图片路径是否可访问。
- 代码块是否有语言标记。
- 手机阅读段落是否过长。
- 封面标题是否清楚、无错字。
- 是否存在多余草稿、临时文件或无用说明。

## 推荐文件命名

```text
<文章名>-大纲.md
<文章名>-初稿.md

<文章名>-审校.md

## 协作边界

- `technical-blog-writing` 可以创建 Mermaid 源码，但不负责转图片。
- `huashu-proofreading` 可以修改文章结构和表达，但不负责生成封面。
- `markdown-mermaid-to-images` 只在定稿后运行，避免反复返工。
- `wechat-cover-generation` 只基于最终标题和主题做封面，不反向改正文。
