## 排查线上问题思路

复现 -> 打点/录制 -> 拆阶段 -> 定位瓶颈 -> 验证优化

“如果问题集中在‘展开课程目录’，我会先稳定复现这个交互，然后从监控平台确认是这类交互导致 INP 抬高，再到浏览器性能面板录一次交互过程。
定位上我会重点拆三段：
第一，点击后到事件真正开始执行之前，主线程是不是已经被别的任务占满；
第二，展开逻辑本身有没有同步重计算，比如目录树递归处理、复杂 JSON 转换、权限过滤、排序聚合；
第三，状态更新后有没有触发大范围重渲染，比如一次展开导致整棵目录树重新渲染、布局重算或大量 DOM 挂载。
像课程目录这种场景，我会优先怀疑大列表或树组件全量渲染、递归组件层级深、响应式依赖范围过大、展开动画叠加布局计算、以及第三方埋点或播放器脚本在交互时抢占主线程。
如果确认问题存在，我通常会优先尝试按需渲染子节点、做分片渲染或虚拟化、收窄状态更新粒度、把非关键计算移到空闲时机执行。””


## 数据是怎么得来的

“这个收益我们不是只看一次前后对比，而是尽量做了统一口径验证。首先，2.1 秒和 0.5 秒都是同一套性能监控口径下统计的同类页面数据，不是随便拿两个时间点对比。其次，SSR 是灰度上线的，我们在灰度阶段对比了新旧链路的 TTFB、首屏时间和服务端 render 耗时，确认提升主要集中在 SSR 相关链路上。再加上缓存命中率提升、实时渲染请求占比下降，这些过程指标也能说明优化确实生效了。因为同期也有接口字段精简和资源优化，所以我们不会说 0.5 秒完全是某一个点带来的，而是这套 SSR 架构优化整体产生的结果。”

“我们做过灰度对照，不是全量 A/B 实验。灰度阶段同一类页面下，新链路相对于旧链路在 TTFB 和首屏时间上都有明显下降，等稳定后再全量上线。”


灰度发布：
1. 用户请求 -> 网关/CDN边缘节点 -> 分流决策层 -> 服务与资源层
2. 分流决策层：
  - 观测与执行层：监控系统（错误率/LCP/INP）超过阈值 -> 自动触发回滚
  - 规则引擎：用户 ID 哈希/地域/白名单
3. 服务与资源层：
  - 存储与配置：灰度规则存储、动态配置中心 -> 新版本 SSR 集群（金丝雀环境）/ 稳定版 SSR 集群

金丝雀环境是软件部署中的一个形象比喻，来源于历史上的"矿井金丝雀"做法——矿工曾携带金丝雀下井，因为金丝雀对有毒气体极其敏感，一旦金丝雀停止鸣叫或死亡，矿工就立刻撤离，从而避免灾难。
金丝雀环境 = 小规模、真实流量、先于全量上线的新版本运行环境
具体来说，就是将新版本先部署到一小部分真实用户面前，让这部分"金丝雀用户"先体验新版本，通过监控他们的反馈和系统指标，来判断新版本是否稳定，再决定是否逐步扩大范围直至全量上线。
