**项目难点：分布式环境下高可靠做题数据同步系统的设计与实现**

作为在线教育平台的核心技术负责人，我设计并落地了一套支撑千万级用户、日均百万次答题的高可靠数据同步系统，解决了分布式环境下数据一致性、可用性和性能的平衡难题。

**业务痛点与挑战**：

随着业务规模扩大，传统"请求-响应"式数据保存方式暴露出严重问题：
- **数据丢失**：弱网、崩溃导致5%的做题记录丢失，引发大量用户投诉
- **并发冲突**：多端同时答题产生数据不一致，用户在不同设备看到不同的学习进度
- **性能瓶颈**：高频保存请求导致服务器压力过大，高峰期响应延迟高达3秒
- **体验割裂**：设备切换后需要重新答题，学习连续性差

**技术方案与创新**：

- 确立分支与合并规范
- 规划多环境策略
  - 开发环境 (Development)：开发者推送任意功能分支时自动触发，用于快速验证。
  - 预发布/测试环境 (Staging/Test)：当合并请求被合入 develop 或 main 分支时触发，用于集成测试和产品验收。
  - 生产环境 (Production)：仅当创建特定格式的标签（如 v1.2.3）或手动点击运行时触发，需要严格控制，确保最高稳定性。

- 管理敏感信息 —— 安全无小事
  - 配置位置：进入项目页面，点击 Settings > CI/CD > Variables。
  - 添加变量：点击 "Add variable"，将你的 SSH_PRIVATE_KEY、PROD_SERVER_IP、NPM_TOKEN 等键值对添加进去。
  - 关键设置：对于敏感信息，务必勾选 Mask variable。这样，变量值会在作业日志中被隐藏，防止泄露。不过请注意，被 Mask 的变量值需要满足一定的格式要求（如长度、字符集），SSH 私钥通常无法被 Mask，但可以通过将其存储在文件中的方式间接使用。

2. SSR是如何处理 Node.js 服务端的稳定性问题的？包括健康检查、限流限频
- 健康检查
  - 活性探针
  - 就绪探针
  - 自定义指标探针：还可以基于业务指标（如最近一分钟的请求错误率、CPU 或内存阈值）来判断实例的健康状态，实现优雅的下线
- 限流与限频 —— 为流量安装"智能水龙头"
  - 静态资源 (.js, .css, .png)：可以直接在 CDN 或 Nginx 层限流，压力较小。
  - API 数据接口：处理轻量级数据请求，可以设置较高的阈值 。
  - SSR 页面接口：渲染复杂页面，资源消耗大，必须设置更严格的阈值。
  - 限流算法选型
    - 固定窗口：实现简单，但存在窗口边界流量突发的问题。
    - 滑动窗口：比固定窗口更平滑，能更精确地控制流量 。
    - 令牌桶：强烈推荐用于 SSR 服务。它允许一定程度的突发流量（应对瞬间热点），又能保证长期的速率限制，非常符合用户访问的波动特性 
  - 限流实践方案
    - 方案一：中间件层限流
    ```js
    const express = require('express');
    const rateLimit = require('express-rate-limit');
    const app = express();

    // 严格限流器：针对 SSR 页面渲染，每个 IP 每分钟最多 30 次请求
    const ssrLimiter = rateLimit({
      windowMs: 60 * 1000, // 1 分钟
      max: 30, // 限制每个 IP 在 windowMs 内最多 30 个请求
      message: '请求过于频繁，请稍后再试。', // 触发限流时的返回信息
      statusCode: 429, // 返回 Too Many Requests 状态码 [citation:9]
      keyGenerator: (req) => req.ip, // 以 IP 作为识别 key
      standardHeaders: true, // 返回 RateLimit-* 头信息
      legacyHeaders: false,
    });

    // 宽松限流器：针对 API 数据接口
    const apiLimiter = rateLimit({
      windowMs: 60 * 1000,
      max: 100,
      message: 'API 请求频率过高',
      statusCode: 429,
    });

    // 应用限流器到特定路由
    app.use('/product/*', ssrLimiter); // 商品详情页这类 SSR 路由
    app.use('/api/', apiLimiter);       // 数据 API 路由
    ```

    - 方案二：基础设施层限流
      - Nginx 层限流：利用 Nginx 的 limit_req 模块，在流量到达 Node.js 之前就进行拦截。可以基于 $binary_remote_addr 进行限流。这种方式性能最好，且不占用 Node.js 进程资源。
      - 网关层限流：如果你的架构使用了 API 网关（如 Kong、Traefik 或云服务商的 API 网关），可以在网关配置统一的限流策略，甚至针对不同用户（API Key）设置不同的配额

## 什么情况下会服务降级？
降级分为被动降级和主动降级：
1. 被动降级
  - Node.js 服务进程崩溃
  - 渲染js报错
  - 渲染超时。为每个页面的渲染过程设置一个全局超时时间（例如 2 秒）。一旦超时，服务端立即中断渲染，返回一个包含 Loading 状态的 CSR 模板。这能有效防止因单个页面慢而耗尽所有服务线程。
  - 接口依赖失败
  - 内存使用率过高。SSR 是 CPU 和内存密集型操作。如果由于代码问题（如全局缓存不当导致的内存泄漏），Node 进程的内存使用率超过阈值（如 90%）。
2. 主动降级 
  - 触发限流策略。遇到突发流量（如营销活动），请求量超过了我们设定的阈值（例如 express-rate-limit 每 IP 每分钟 30 次，或整体 QPS 超过 1000）。
  - 爬虫与低性能客户端识别
