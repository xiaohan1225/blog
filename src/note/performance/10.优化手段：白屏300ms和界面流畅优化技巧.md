## 引言

在性能监控当中，白屏时间和卡顿也是两个值得注意的优化指标。

白屏时间越短，给人感觉 App 速度快，体验好。

卡顿无疑对业务转化率有很大的影响。

## 白屏优化

在某电商 App 上搜索后进入商品列表页，结果屏幕一片空白，过了好久还是没什么内容出现，这时候你是不是会退出来，换另一个电商 App 呢？

首先，在性能平台上查看白屏时间指标，确认是否是白屏问题，问题确认后，我们可以基于影响白屏时间长短的两个主要因素来解决--DNS 查询和首字符展示。

## DNS 查询优化

DNS 查询是指浏览器发起请求时，需要将用户输入的域名地址转换为 IP 地址的过程，这个转换时间长短就会影响页面的白屏时间。

```js
// 获取当前页面的性能数据
const performanceData = window.performance.timing;

// DNS 解析时间 = domainLookupEnd - domainLookupStart
const dnsTime = performanceData.domainLookupEnd - performanceData.domainLookupStart;

console.log(`DNS 解析耗时: ${dnsTime}ms`);
```

可以从前端和客户端这两部分采取措施。

- 前端侧：可以通过在页面中加入 dns-prefetch，可以减少 150ms 的 DNS 解析时间。

```html
<!-- 开启 DNS 预解析功能 -->
<meta http-equiv="x-dns-prefetch-control" content="on"> 
<!-- 强制对 api.example.com 的域名做预解析 -->
<link rel="dns-prefetch" href="https://api.example.com">
```

- 客户端侧：可以在启动 App 时，同步创建一个肉眼不可见的 Webview（1 x 1px）。将常用的静态资源路径写入到这个 webview 中，然后对它域名解析并放入缓存中，这样后面需要使用 webview 打开真正所需页面时，由于已经做过域名解析了，客户端直接从缓存中获取即可。

如果是端外页面可以使用 iframe，也可以达到类似效果。


它可以将 DNS 解析时间控制在 400ms 以内。

如果你想将请求进一步压缩，比如控制在 200ms，此时就需要一个重量级的方案，具体来说，可以采用 IP 直连的方式，原来是请求 `www.google.com`，现在我们可以通过调用 SDK 进行域名解析，拿到对应的 IP（如 6.6.6.6），然后直接请求这个 IP 地址拿到数据。

这个实现起来的坑：
- HTTPS 证书和配置文件。HTTPS 证书是指当客户端使用 IP 直连时，请求 URL 中的 host 会被替换成对应的 IP。所以在证书验证时，会出现 domain 不匹配的情况，导致 ssl 或 tls 协商不成功。

如何解决呢？

- 在非 SNI 的场景下，可以把证书验证环节独立出来，如 hook 证书校验环节，然后将 IP 替换为原来的域名。
- 在 SNI 场景下，可以定制 SSLSocketFactory，在 createSocket 时替换为 IP，并进行 SNI/HostNameVerify 配置。
- 配置文件方面，一般在域名只有两三个的情况时，我们可以用到它来做 IP 和域名的映射。但是随着机房的扩大，每次机器都要升级配置文件，后续会非常麻烦，对此我们可以采用 httpDNS 来解决。httpDNS 可以准确调度到对应区域的服务端 IP 地址给用户，同时还可以避免运营商 DNS 劫持。具体来说，SDK 会通过发报文向 httpDNS 做一个 http 请求，通过后拿到对应域名，然后进行 IP 直连，完成资源或接口的请求。

## 首字符展示优化

通常我们会在页面加载过程中出现一个 loading 图，用来告诉用户页面内容需要加载，请耐心等待。

但一个 loading 既无法让用户感觉到页面加载到什么程度，也无法给用户直觉上一个焦点，让人们的注意力集中在上面，如何解决这个问题？

- 骨架屏（Skeleton Screen）是指在页面数据加载完成前，先给用户展示出页面的大致结构（灰色占位图），告诉用户页面正在渐进式地加载中，然后在渲染出实际页面后，把这个结构替换掉。
- 骨架屏没有真正减少白屏时间，但是给了用户一个心里预期，让他可以感受到页面上大致有什么内容。

如何构建骨架屏呢？
- 每次视觉修改或者功能迭代，骨架屏都要配置修改，我建议采用**自动化方案**，而不是手动骨架屏方案。

骨架屏的实现步骤：
1. 确定生成规则，遍历所有的 DOM 元素。针对特定区块（如视频、音频）生成的对应的代码块，获取原始页面中 DOM 节点的宽度、高度和距离视窗的位置，计算出当前设备宽高对应的大小，转换成相应的百分比，然后来适配不同的设备。
2. 基于上述规则，结合 CLI 工具可以通过脚手架自动生成骨架屏。
3. 将骨架屏自动化注入页面，再利用 Puppeteer 把骨架屏代码注入页面中自动运行。

这个实现过程比较复杂有很多坑，具体在 14 讲中介绍。

## 卡顿治理

卡顿现象一般可以通过用户反馈或性能平台来发现。

在性能平台上查看卡顿指标后，发现页面出现连续 5 帧超过 50ms，这就属于严重卡顿。

- 首先是问题的定位
- 通过 charles 等工具抓包看一下数据接口，如果和数据相关的问题，找后端同事或者用数据缓存的方式解决。
- 前端侧问题，一般是**浏览器的主线程和合成线程调度不合理**，以及**计算耗时操作**。

浏览器的主线程和合成线程调度不合理：
- 在红包位置变化时。页面展现时特别卡，这就是主线程和合成线程调度的问题，怎么解决呢？

-主线程主要负责运行 JavaScript，计算 CSS 样式，元素布局然后交给合成线程，合成线程主要负责绘制。当使用 height、width、margin、padding 等作为 transition 的值时，会让主线程的压力很大，可以使用 transform 来代替直接设置 margin 等操作。比如红包元素从 `margin-left: 10px` 渲染到 `margin-left: 0`，主线程需要计算样式 `margin-left: 9px，margin-left: 8px` 一直到 `margin-left: 0`，每一次主线程计算样式后，合成线程都需要绘制到 GPU 再渲染到屏幕上，来来回回需要进行 10 次主线程渲染。这样会给浏览器带来很大压力，导致卡顿。
- 我们可以利用 transform 来做，比如 `transform: translate(-10px, 0)` 到 `transform: translate(0, 0)`，主线程只需要进行一次 `transform: translate(-10px, 0)` 到 `transform: translate(0, 0)`，然后合成线程去一次将 -10px 转换到 0px。这样的话可以减少 9 步操作，假设一次是 10ms，就可以减少 90 ms。

计算耗时操作的解法：
- 空间换时间

在对 DOM 元素增删的过程中最好先在 DocumentFragment 上操作，而不是直接在 DOM 上操作。只在所有操作更新完成后，将 DocumentFragment 的变动更新到 DOM 上，从而解决频繁更新 DOM 带来的卡顿问题。

- 时间换空间。

一般是通过将一个复杂的操作细分成一个队列，然后通过多次操作解决负责操作的问题。比如一个 WebIM 离线消息的项目，其中一个功能是将批量消息下载到本地，点击下载以后由于文件过大，浏览器出现卡顿甚至假死的情况。

最后是通过实现一个 chunk 的方式，创建了一个队列，定时取出队列的任务，然后在本地服务器将内容再次合并的方式来解决。从而解决复杂任务带来的卡顿问题。

```js
function chunk(array, process, content) {
  setTimeout(function () {
    var item = array.shift();
    process.call(context, item);
    if (array.length > 0) {
      setTimeout(arguments.callee, 100);
    }
  }, 100)
}
```

## 小结
setTimeout 设置定时时间不准，比如设置 0ms，实际最小要 4ms，如果间隔时间过小，比如小于 16ms，还会出现因为掉帧导致的卡顿，幸好新版本浏览器 IE10 以上提供了 `requestionAnimationFrame` 方法较好来解决它？
