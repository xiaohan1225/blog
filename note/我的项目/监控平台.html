项目名称： BT一站式前端监控平台

  • 角色： 前端开发组长
  • 开始时间：2023-06

项目描述：为了解决线上问题“黑盒”现象及用户报障难以复现的问题，主导设计了公司内部的前端监控SDK及可视化平台。实现了对JavaScript错误、资源加载失败、API请求异常、白屏以及用户核心行为轨迹的采集与报警，覆盖了公司主站及后台管理系统共 10+ 个核心业务，日均处理百万级日志数据。

技术栈：Vue3、TypeScript、Vite、Next.js、Antv、Kafka、ClickHouse 等。

个人职责： 1、核心 SDK 开发：从 0 到 1 设计轻量级监控 SDK（压缩后 <30KB），采用插件化架构，支持错误、性能、行为埋点等能力按需启用，并通过统一上下文、事件总线和批量上报机制降低对业务代码的侵入。 2、性能监控：基于 Performance API 和 Web Vitals 采集 FCP、LCP、INP、CLS 等核心指标，建立页面性能基线；结合业务场景补充秒开率、业务关键区域渲染时间、卡顿等指标，支撑体验治理。 3、异常监控：设计统一异常上报体系，覆盖前端 JavaScript 运行时错误、资源加载失败、API 请求异常、自定义业务异常等场景，并结合 SourceMap 实现错误源码定位。 4、数据可视化与分析：基于 ClickHouse 搭建海量日志存储与查询层，结合 Vue3 + AntV G2 设计总览、错误分析、性能分析、用户行为路径等核心页面，帮助研发从“发现问题”快速走到“定位根因”。

项目难点：

1、首屏加载时间难以量化，广义白屏难以精准采集。通过设计基于 MutationObserver 的首屏检测算法，结合骨架屏过滤、关键资源加载与页面稳定窗口判定，将首屏时间检测准确率提升至 95% 以上，并能识别接口、数据加载、图片/视频资源导致的白屏问题。 2、性能预警容易演变为告警风暴，导致系统处理延迟和“告警疲劳”。通过引入告警抑制与降噪机制，实施告警分级与分渠道处理、冷热数据分离、限流与熔断等方案，保障监控系统在高噪声场景下的可用性与告警有效性。 3、海量日志下的采样与计算成本较高。针对性能数据在前端 SDK 侧设计动态采样策略，并在服务端进行实时聚合计算，生成 Web Vitals 趋势与分层分析结果，在保证统计代表性的同时控制计算与存储成本。

项目成果： 1、覆盖 95% 以上核心页面，解决过去线上问题“黑盒”状态，故障发现时效从用户反馈的小时级缩短至系统告警的分钟级。

2、基于监控平台对做题流程相关接口异常、超时分布和用户行为路径的关联分析，定位到网络抖动场景下关键链路稳定性问题；推动前后端协同优化超时策略、可恢复错误重试、提交态保护、接口幂等与失败兜底机制后，相关场景下做题流程异常率从 5% 降至 0.3%。 3、结合 SourceMap 实现错误源码定位，将线上 JS 异常排查从“依赖用户复现”缩短到“直接定位源码文件与代码行”，前端团队 Bug 排查效率提升 60%。 4、建立页面体验评级模型，基于监控平台发现课程详情页 LCP 高达 3.8s，并在可视化平台拆解资源加载与渲染耗时；通过懒加载、接口优化、渲染策略优化和图片方案升级等手段，使 LCP 中位数从 2.8s 降至 1.6s，并结合 A/B 测试持续监控“选课-下单-支付”核心漏斗页面，优化后订单转化率提升 3%。

这是一个非常典型的“告警风暴”问题，不仅会影响监控系统的性能，还会导致接收邮件的人产生“告警疲劳”，最终忽视真正重要的预警。

针对你提到的**“预警邮件越来越多，后台拖慢，只能展示2小时前的数据”**这一现象，核心矛盾在于：写数据（产生告警日志）的速度远大于读数据（展示、处理告警）的速度，且后台处理逻辑存在瓶颈。

为了解决这个问题，通常需要从**“告警抑制”、“分级处理”和“系统架构解耦”**三个维度入手。以下是具体的解决思路和方案：

1. 引入“告警抑制”与“降噪”机制 ​

既然性能问题短期内无法解决，就不能让它无休止地报警。需要让系统变得“智能”一些，减少重复劳动。

  • 状态翻转机制（Flapping）：
    • 问题： 每次检测到性能超标就发邮件，问题恢复又发邮件，反复波动。
    • 解决： 引入类似监控系统（如Zabbix/Prometheus）的机制。只有当问题持续一定时间（例如持续5分钟）或连续发生N次探测失败后，才触发警报。问题恢复后，24小时内不再重复发送相同原因的邮件。
  • 沉默周期/告警聚合：
    • 问题： 每5分钟跑一次脚本，每次跑出100个慢SQL，就发100封邮件。
    • 解决： 改为将同一时间段内（如30分钟内）的同类错误合并为一封邮件，标题注明“有N个接口发生超时”，而非逐一发送。

2. 实施告警分级与分渠道处理 ​

并不是所有的预警都需要通过“邮件”这种重成本的方式通知，也并不是所有预警都需要立即展示在前端。

  • 分级策略：
    • P0/P1（紧急）： 短信/电话，立即处理。
    • P2/P3（警告）： 邮件，但无需立即展示在前端实时看板，可归档。
    • P4（信息）： 仅入库，供排查问题时查询。
  • 为什么这样做： 你提到后台只能展示2小时前的数据，说明查询压力很大。如果将大量P3、P4的日志和预警都塞给前台展示，必然拖慢速度。可以考虑只在前端展示紧急的实时数据，历史数据通过“查询”功能获取。

3. 系统架构解耦 ​

你提到后台拖慢，大概率是因为预警产生、邮件发送、数据查询这三者耦合在一起，互相争抢资源。

  • 引入消息队列：
    • 现状： 脚本检测到问题 -> 直接写数据库 -> 直接发邮件。
    • 优化：
      1. 脚本检测到问题后，将告警内容丢进消息队列，立即返回，不阻塞后续检测。
      2. 由单独的消费者进程从队列中拉取数据，负责写入数据库。
      3. 由邮件服务消费者专门负责发送邮件。
    • 好处： 即使邮件发送卡住（比如SMTP服务器响应慢），也不会影响后台写入数据库的速度；即使数据库写入慢，也不会影响检测脚本的运行。
  • 读写分离：
    • 预警写入数据库（写库）和前端查询展示（读库）分离。
    • 写入的数据通过binlog同步或定时同步到读库。
    • 即使查询很复杂，拖慢了读库，写库依然可以快速接收新的预警，保证数据不积压。

4. 冷热数据分离 ​

你提到只能展示2小时前的数据，说明数据库里数据量太大了，查询变慢。

  • 分表/分区： 按天甚至按小时分区。查询2小时内的数据，只需要扫描最近的分区，速度会很快。
  • 数据归档： 将超过24小时或48小时的预警数据，自动转移到归档表或归档数据库（甚至是大数据平台）。前端默认只查询热数据，需要看历史数据时，需要加时间筛选并查询慢一些的归档库。

5. 限流与熔断 ​

在极端情况下，需要保护监控系统自身。

  • 丢弃策略： 如果同一时刻产生大量重复预警（比如机房断电，上万台机器同时报失联），系统应该能够识别并丢弃非核心的告警，只发送一条“大面积服务器失联”的汇总通知。

总结建议 ​

针对你当前的情况，建议按以下优先级处理：

  1. 立即止血： 修改发邮件的逻辑，增加**“沉默时间”**。如果同一个IP或同一个接口在最近1小时内已经报过错，则不再发送邮件（或仅增加计数），这能立即减少90%以上的邮件量和后台IO压力。
  2. 恢复性能： 对告警表进行数据库分区（按小时），删除或归档2天前的数据，让前台查询只针对小数据集，尽快恢复到实时查看。
  3. 长期改造： 引入消息队列，实现告警产生、存储、发送的解耦。

这样做下来，预警系统就不会因为底层业务的性能问题而被拖垮，也能更好地发挥预警作用。
