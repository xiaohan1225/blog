BT低代码平台 ​

BT低代码平台是一个面向内部运营的自研低代码平台，支持营销活动页和落地页的可视化搭建、发布和版本管理，累计支撑了 300+ 线上页面交付，简单页面交付周期从原来的 2 周 缩短到 2-3 天。

我在这个项目里担任前端负责人角色，主要负责三块：

  • 第一块是平台架构设计，采用 Monorepo + 微内核 + 插件化，把 Schema 协议、插件注册机制、渲染上下文、命令系统 这些稳定的平台级通用能力沉淀到内核里，把 物料、属性设置器、数据源适配、调试面板、发布扩展 这些高频变化能力做成插件，保证平台可持续扩展。
  • 第二块是核心引擎建设，包括 Schema、Renderer、拖拽编排引擎。Schema 不是普通组件树 JSON，而是平台内部的统一协议，结构关系和节点详情分离，方便局部更新、调试和历史回放；Renderer 做成协议驱动的通用执行层，新增物料主要靠注册，不需要频繁改渲染内核；拖拽这块重点解决复杂嵌套布局下的目标容器命中和放置位置判断，采用 射线检测 + 最近父容器回溯，并通过缓存几何信息、requestAnimationFrame 合帧、局部更新和滚动增量修正提升拖拽性能。
  • 第三块是表达式和调试体系，基于 CodeMirror 6 做变量编辑器，支持补全、错误提示和上下文感知，运行时通过受控上下文和沙箱限制危险能力，同时设计了业务层、实施层、技术层三层调试能力，帮助定位表达式、数据绑定和渲染链路问题。

前端全链路监控平台 ​

“这个是企业内部的一个前端全链路监控平台，主要解决的是线上问题黑盒、用户报障难复现、排障效率低的问题。我主导设计并落地了公司内部的前端监控 SDK 和可视化分析平台，把前端侧从用户访问、页面渲染、用户行为、接口请求、异常上报到告警分析和问题定位这整条链路打通，覆盖了公司 10 多个核心业务，日均处理百万级日志数据。

在这个项目里，我主要负责三块：

  • 第一是从 0 到 1 设计轻量级监控 SDK，采用插件化架构，统一采集 JS 错误、资源异常、接口异常、性能指标和行为轨迹；
  • 第二是性能和白屏监控，基于 Web Vitals 和 MutationObserver 设计了秒开率、首屏检测、业务关键区域渲染时间等指标；
  • 第三是可视化分析平台建设，基于 ClickHouse 做海量日志分析，并结合 SourceMap 实现错误源码定位。

项目落地后，覆盖了 95% 以上核心页面，故障发现时效从用户反馈的小时级缩短到系统告警的分钟级，线上 JS 异常排查效率提升了 60%，同时还基于监控数据持续推动业务优化，比如把网络抖动场景下做题流程异常率从 5% 降到了 0.3%。”

排查线上问题思路 ​

复现 -> 打点/录制 -> 拆阶段 -> 定位瓶颈 -> 验证优化

“如果问题集中在‘展开课程目录’，我会先稳定复现这个交互，然后从监控平台确认是这类交互导致 INP 抬高，再到浏览器性能面板录一次交互过程。 定位上我会重点拆三段： 第一，点击后到事件真正开始执行之前，主线程是不是已经被别的任务占满； 第二，展开逻辑本身有没有同步重计算，比如目录树递归处理、复杂 JSON 转换、权限过滤、排序聚合； 第三，状态更新后有没有触发大范围重渲染，比如一次展开导致整棵目录树重新渲染、布局重算或大量 DOM 挂载。 像课程目录这种场景，我会优先怀疑大列表或树组件全量渲染、递归组件层级深、响应式依赖范围过大、展开动画叠加布局计算、以及第三方埋点或播放器脚本在交互时抢占主线程。 如果确认问题存在，我通常会优先尝试按需渲染子节点、做分片渲染或虚拟化、收窄状态更新粒度、把非关键计算移到空闲时机执行。””

项目难点：

发现问题 -> 复现/排查问题 -> 解决问题 -> 验收问题 -> 优化发现问题、排查问题流程，怎么保证优化后不会出新问题，上线后如何验证优化效果 -> 反思（在做项目之前，先考虑好项目的风险性，考虑用户体验）

  1. 发现问题 -> 用户反馈说做题记录记录丢失

  2. 复现/排查问题 -> 通过日志分析，某天学员有开卷记录，但没有做题记录，进一步分析发现，有两种情况导致了做题记录丢失，第一种是弱网和意外退出，第二种是多端做题下导致的数据覆盖问题。

  3. 解决问题 -> 针对弱网，增加本地缓存、重试、提交态保护机制，针对意外退出，增加钩子拦截，针对多端做题记录覆盖问题，我们引入了 CRDT 数据结构以及事件溯源机制，可以理解为“带自动冲突解决的 Git”，确保在多端做题下，做题记录过程可追踪、可恢复，能够正确合并。

  4. 如何验证

  5. 手动模拟弱网、意外退出、多端做题场景，验证做题记录是否能够正确合并。

  6. 验证在多端做题下，做题记录是否能够正确合并。

  7. 如何判断优化前后是否有效果？ - 引入做题丢失率指标，验证优化后做题记录丢失率是否下降。

  8. 怎么保证优化后不会出新问题? - 通过灰度发布，先在小范围内验证，确保新版本在新场景下能够正常运行。

  9. 反思

  • 如何提前发现问题？ -> 在原来前端错误监控的基础上么，增加指标统计和指标告警机制
  • 如何快速排查问题？ -> 建立做题记录统计平台，统计用户的网络情况，事件溯源完成流程。

数据是怎么得来的 ​

“这个收益我们不是只看一次前后对比，而是尽量做了统一口径验证。首先，2.1 秒和 0.5 秒都是同一套性能监控口径下统计的同类页面数据，不是随便拿两个时间点对比。其次，SSR 是灰度上线的，我们在灰度阶段对比了新旧链路的 TTFB、首屏时间和服务端 render 耗时，确认提升主要集中在 SSR 相关链路上。再加上缓存命中率提升、实时渲染请求占比下降，这些过程指标也能说明优化确实生效了。因为同期也有接口字段精简和资源优化，所以我们不会说 0.5 秒完全是某一个点带来的，而是这套 SSR 架构优化整体产生的结果。”

“我们做过灰度对照，不是全量 A/B 实验。灰度阶段同一类页面下，新链路相对于旧链路在 TTFB 和首屏时间上都有明显下降，等稳定后再全量上线。”

灰度发布：

  1. 用户请求 -> 网关/CDN边缘节点 -> 分流决策层 -> 服务与资源层
  2. 分流决策层：
  • 观测与执行层：监控系统（错误率/LCP/INP）超过阈值 -> 自动触发回滚
  • 规则引擎：用户 ID 哈希/地域/白名单
  1. 服务与资源层：
  • 存储与配置：灰度规则存储、动态配置中心 -> 新版本 SSR 集群（金丝雀环境）/ 稳定版 SSR 集群

金丝雀环境是软件部署中的一个形象比喻，来源于历史上的"矿井金丝雀"做法——矿工曾携带金丝雀下井，因为金丝雀对有毒气体极其敏感，一旦金丝雀停止鸣叫或死亡，矿工就立刻撤离，从而避免灾难。 金丝雀环境 = 小规模、真实流量、先于全量上线的新版本运行环境 具体来说，就是将新版本先部署到一小部分真实用户面前，让这部分"金丝雀用户"先体验新版本，通过监控他们的反馈和系统指标，来判断新版本是否稳定，再决定是否逐步扩大范围直至全量上线。
