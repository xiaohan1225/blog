三、高频面试清单 A. SSR 性能优化 为什么要做 SSR，原来慢在哪里 关键词：TTFB、接口聚合、SSR render、HTML 返回、hydrate

你们是怎么定位瓶颈的 关键词：前端监控 + 服务端日志/APM + 链路拆解

三层缓存分别是什么，各自缓存什么 关键词：CDN HTML/静态资源、Redis 聚合数据/部分 HTML、Node 热点缓存

缓存怎么失效 关键词：TTL + 主动失效 + ISR

流式渲染和 ISR 分别解决什么问题 关键词：流式渲染解决返回时机，ISR 解决实时 SSR 成本

为什么两个都要用 关键词：一个解决“返回太晚”，一个解决“算得太多”

你怎么证明 2.1s 到 0.5s 是这套方案带来的 关键词：统一口径、灰度对照、TTFB/render 耗时下降、缓存命中率提升

B. 微前端改造 为什么要做微前端，不是继续修修补补 关键词：多历史项目并存、技术栈异构、公共能力分散、统一入口、渐进迁移

为什么选 qiankun，不是 iframe / Module Federation / Wujie 关键词：场景匹配度、成熟度、接入成本、异构系统整合

接入 10+ 历史项目时，你们最先定的规范是什么 关键词：打包输出、publicPath、生命周期、路由 base、样式隔离、通信协议、监控接入

样式冲突怎么解决 关键词：作用域隔离、限制全局样式、弹层挂载容器

路由冲突怎么解决 关键词：独立路由前缀、activeRule、独立运行/接入基座双态

应用间通信怎么做，为什么不用一个大 store 关键词：最小必要共享、initGlobalState、主应用上下文、业务状态留子应用

微前端最大的坑是什么 建议固定答：子应用卸载不干净导致副作用残留

这个坑最后怎么规模化治理 关键词：接入清单、统一封装、生命周期绑定、验收门禁、灰度观察

C. 答题同步架构 为什么要用事件溯源 + CRDT 关键词：本地缓存+重试只能提成功率，解决不了并发冲突

两者分别解决什么问题 关键词：事件溯源负责过程可追踪/恢复，CRDT 负责并发合并

哪些数据用了 CRDT，哪些没有 关键词：主观题文本/草稿用 CRDT；客观题、提交状态、判题结果不用

增量同步协议怎么设计 关键词：clientId/deviceId、update 日志、lastAck/checkpoint/state vector

snapshot、update 日志、checkpoint 分别是什么 关键词：完整状态、增量记录、同步位点

幂等和去重怎么做 关键词：Yjs update 幂等 + 协议层 ack/seq

分叉恢复怎么做 关键词：轻度增量补齐，中度回放缺失 update，严重回退 snapshot

数据丢失率怎么定义 关键词：本地最后有效状态未体现在服务端最终恢复/提交结果里，且非用户主动修改

弱网答题成功率怎么定义 关键词：用户视角下最终答案是否保住，不是请求成功率

多端同时编辑后又提交，最终答案怎么裁决 关键词：编辑态可合并，提交态由服务端定版

D. 工程效能 / 研发治理 Code Review 自动化做了什么 关键词：规则沉淀、CI 接入、风险扫描、门禁化

Vue2 升 Vue3 为什么能渐进推进 关键词：分模块迁移、兼容层、风险分级、灰度验证

AI 能力在项目里怎么落地 关键词：评审辅助、测试增强、需求拆解、工程提效

如何证明效能提升不是主观感受 关键词：覆盖项目数、测试覆盖率、交付效率、线上问题率

四、每条主线的核心金句 SSR 流式渲染解决的是返回太晚，ISR 解决的是实时 SSR 成本高。 缓存的本质不是提速一层，而是减少重复取数和重复渲染。 微前端 微前端解决的是系统边界和组织协同问题，不是代码质量本身。 不是不共享状态，而是最小必要共享，避免退化成分布式巨石。 答题同步 传统方案解决上传成功率，不解决并发一致性。 事件溯源负责可恢复，CRDT 负责可合并。 工程效能 效能建设不是加工具，而是把规范做成流程和门禁。 五、建议你的答题顺序 面试里如果让你自由介绍这个项目，建议顺序固定成：

先一句话交代项目规模和你的角色 重点讲 SSR 再讲 微前端 再讲 答题同步 最后补 工程效能 这个顺序最好，因为：

SSR 最容易先建立技术深度 微前端 最能体现架构和治理能力 CRDT 最能拉开辨识度 效能 适合最后补负责人视角 六、你要避免的坑 不要把 SSR 和 SPA/single-spa 混着讲 不要把 qiankun 沙箱 说成完全解决样式冲突 不要把 initGlobalState 说成“完全没有共享 store” 不要把 CRDT 说成所有答题数据都在用 不要把 “性能提升/成功率提升” 说成没有口径定义的绝对数字

3 分钟面试背诵稿 ​

可以，下面这版是更适合面试口述的 3 分钟背诵稿，我按“项目背景 + 我的角色 + 4 个核心成果”帮你压缩好了：

“BT 教育平台是一个覆盖 PC 官网、管理后台、H5、Hybrid 和小程序的综合教育平台，业务包括课程体系、直录播、在线做题、支付、营销和社区等。我在这个项目里担任的是前端负责人角色，主要负责前端架构设计、核心技术方案落地，以及工程效能建设。

这段经历里我重点做了四件事。

第一件是官网 SSR 性能优化。最开始用户反馈官网首页、课程详情页这类高流量页面首屏比较慢，我们结合前端监控和服务端链路，把首屏拆成 TTFB、接口聚合、SSR render、HTML 返回、首屏渲染和 hydrate 几段来看，最后发现核心瓶颈主要在接口聚合耗时、实时 SSR 重复计算，以及整份 HTML 生成完才返回导致首字节偏晚。后面我们做了三层缓存，分别是 CDN 边缘缓存、Redis 分布式缓存和 Node 本地热点缓存，同时结合流式渲染和 ISR，分别解决“返回太晚”和“实时 SSR 成本高”的问题，最后把首屏时间从 2.1 秒降到了 0.5 秒左右。

第二件是微前端改造。当时平台里有 10 多个历史项目并存，技术栈包括 Vue2、Vue3、React 和原生 JS，公共能力分散、系统体验割裂，发布和协同成本也比较高，所以我们基于 qiankun 做了统一基座，把多个历史系统逐步接到同一入口下，同时保留独立开发和独立部署能力。这个过程中我重点推进了接入规范建设，解决了样式冲突、路由冲突、应用通信和子应用卸载不干净这些问题，最后在 3 个月内完成了 10 多个历史项目的渐进接入。

第三件是在线做题场景下的高可靠数据同步。因为弱网、多端切换、意外关闭这些场景下，传统的本地缓存加接口重试只能提升上传成功率，解决不了并发冲突和数据覆盖问题，所以我们设计了事件溯源加 CRDT 的同步架构。事件溯源主要负责作答过程的可追踪和可恢复，CRDT 主要解决主观题文本和草稿区这类数据的并发合并，再配合 snapshot、update 日志和 checkpoint 做增量同步和断点恢复。这个方案落地后，答题记录丢失率从 5% 降到了 0.03%，弱网答题成功率提升到了 99.5%。

第四件是研发效能建设。我推动了 Code Review 自动化、CI/CD 接入、Vue2 到 Vue3 的渐进升级，以及 AI 能力在评审和测试增强中的落地。最终 Code Review 自动化覆盖了 30 多个项目，测试覆盖率从 40% 提升到 80%，核心业务的 Vue3 升级也完成了 80% 左右，整体交付效率和线上稳定性都有明显提升。

如果要总结这段经历，我觉得最能体现我能力的，一方面是能在复杂业务系统里做架构抽象和关键方案取舍，比如 SSR、微前端和高可靠同步；另一方面是能把这些方案真正落到工程治理和业务结果上，而不只是停留在技术设计层面。”
