项目介绍版 ​

BT低代码平台是一个面向内部运营的自研低代码平台，支持营销活动页和落地页的可视化搭建、发布和版本管理，累计支撑了 300+ 线上页面交付，简单页面交付周期从原来的 2 周 缩短到 2-3 天。

我在这个项目里担任前端负责人角色，主要负责三块：

  • 第一块是平台架构设计，采用 Monorepo + 微内核 + 插件化，把 Schema 协议、插件注册机制、渲染上下文、命令系统 这些稳定的平台级通用能力沉淀到内核里，把 物料、属性设置器、数据源适配、调试面板、发布扩展 这些高频变化能力做成插件，保证平台可持续扩展。
  • 第二块是核心引擎建设，包括 Schema、Renderer、拖拽编排引擎。Schema 不是普通组件树 JSON，而是平台内部的统一协议，结构关系和节点详情分离，方便局部更新、调试和历史回放；Renderer 做成协议驱动的通用执行层，新增物料主要靠注册，不需要频繁改渲染内核；拖拽这块重点解决复杂嵌套布局下的目标容器命中和放置位置判断，采用 射线检测 + 最近父容器回溯，并通过缓存几何信息、requestAnimationFrame 合帧、局部更新和滚动增量修正提升拖拽性能。
  • 第三块是表达式和调试体系，基于 CodeMirror 6 做变量编辑器，支持补全、错误提示和上下文感知，运行时通过受控上下文和沙箱限制危险能力，同时设计了业务层、实施层、技术层三层调试能力，帮助定位表达式、数据绑定和渲染链路问题。

如果只选一个最体现架构能力的设计决策，我会选 Schema 协议驱动 + Renderer/物料解耦。因为它把平台从“功能集合”变成了“协议驱动系统”，直接决定了后续扩展性、维护成本和演进上限。

你在简历里写“项目采用 Monorepo + 微内核设计，并通过插件化方式扩展功能”。为什么当时要这么设计？如果直接做一个普通的单体低代码应用，会先遇到什么问题？ ​

“我们当时之所以采用 Monorepo + 微内核 + 插件化，核心是因为低代码平台不是一个功能边界很固定的系统，它天然会不断扩展。比如除了基础画布和渲染能力，后面还会不断增加新物料、属性设置器、事件编排、数据源接入、发布能力、调试工具等。如果一开始就做成普通单体应用，短期开发快，但后面随着功能增多，模块之间会越来越耦合，一个新能力上线经常要改动编辑器、Renderer、Schema、属性面板等多个模块，维护和迭代成本会很高。

Monorepo 主要解决的是工程协同问题。因为平台里其实不止一个应用，而是一组强相关包，比如编辑器、Renderer、物料协议、组件库、工具链、发布模块，它们需要共享类型、Schema 和工具能力。如果拆成多个仓库，版本联动、调试联调、依赖升级都会很重；放在 Monorepo 里，能更好地做依赖管理、类型共享、统一构建和测试，也更适合多人并行开发。

微内核 + 插件化 主要是为了隔离‘稳定内核’和‘高频变化能力’。像 Schema 协议、物料注册、渲染上下文、命令系统、事件总线这些属于平台内核，相对稳定；而像具体业务组件、属性面板、数据源适配器、调试面板、发布扩展这些变化频率高，就做成插件。这样做的好处是，核心能力保持稳定，新增功能尽量通过扩展点接入，不需要频繁侵入内核代码，平台的可维护性和扩展性都会更好。”

那你们当时定义的“内核”到底包含哪些能力？哪些能力必须进内核，哪些能力只能做成插件？ ​

“我们当时判断一个能力要不要进内核，主要看三点：是不是平台级通用能力、是不是多个模块都依赖、以及接口和数据结构是否相对稳定。

所以进内核的，通常是这类能力：第一，Schema 协议和核心数据模型，因为编辑器、Renderer、物料系统、发布链路都依赖同一套描述；第二，插件生命周期和注册机制，因为所有扩展能力都要挂在这个体系上；第三，平台级基础设施，比如命令系统、事件总线、渲染上下文、API 注册表、权限和配置上下文，这些属于多个插件共享的运行基础。

而更适合做成插件的，一般是高频变化、强业务定制、或者可以通过扩展点接入的能力，比如具体物料、属性设置器、数据源适配器、发布目标、调试面板、以及具体的编排节点和动作。 像事件编排这类能力，我们一般会把‘编排引擎和执行框架’放在内核里，把‘具体节点类型和业务动作’做成插件。这样内核保持稳定，业务扩展又不会频繁侵入核心代码。”

你们的 Schema 是怎么设计的？为什么不用直接拿组件树 JSON 就行，为什么还要专门设计数据协议？ ​

“我们当时没有直接把页面存成一棵普通组件树 JSON，而是专门设计了一套 Schema 协议。核心原因是，低代码平台里的页面数据不只是给前端渲染看，还要同时服务编辑器、Renderer、发布链路、版本管理、调试工具和后续扩展，所以它需要是一套稳定、可演进、可校验的数据协议，而不是一个临时的视图结构。

在结构上，我们做了规范化设计，主要分成两部分：一部分是页面结构和节点关系，比如页面根节点、父子关系、节点顺序这些；另一部分是节点详情表，用对象结构按 id 存每个节点的属性、样式、事件、变量绑定、数据源配置等信息，两者通过节点 id 关联。这样做的第一个好处是更新效率高，修改某个节点配置时不需要在整棵树里深度查找，只更新对应节点即可；第二个好处是变更影响范围更小，更适合做局部渲染和增量保存；第三个好处是协议边界更清晰，后面做版本迁移、Schema 校验、历史回放、多人协作或者调试工具时都会更方便。

所以我们设计 Schema，不只是为了存页面，而是为了让整个平台围绕一套统一协议协同工作。普通组件树 JSON 适合简单渲染，但对低代码平台来说，不够稳定，也不够可扩展。”

你们的 Renderer 是怎么和 Schema 解耦的？也就是说，为什么换一套物料或者换一种页面类型，不需要把渲染引擎重写一遍？ ​

“我们当时是把 Renderer 设计成一个协议驱动的通用执行层，而不是写死某一套业务页面。它接收的核心输入不是具体组件实例，而是统一的 Schema、物料注册信息和运行时上下文。 Schema 里只描述页面结构、节点类型、属性、样式、事件和变量绑定；Renderer 自己不关心某个业务组件具体长什么样，它只负责遍历节点、解析节点配置，然后根据 node.type 去物料注册中心找到对应的组件定义和渲染适配器，再把属性、样式、事件、变量结果组装后渲染出来。

这样做的好处是，渲染引擎和具体物料是解耦的。新增一种物料时，只需要按照统一协议注册它的 metadata、渲染组件、默认配置和设置器，不需要改 Renderer 内核。 换页面类型时也是类似的，通常只需要扩展页面容器、布局规则或者运行时上下文，比如 H5 和活动页在容器能力、样式约束、交互规则上可能不一样，但节点遍历、协议解析、变量求值、组件挂载这些核心流程是共用的，所以不需要把渲染引擎重写一遍。

本质上我们做的是三层拆分：Schema 负责描述，物料负责实现，Renderer 负责解释和执行。只要这三层边界清楚，平台扩展时改动就会比较可控。”

1 分钟口语版

“Renderer 我们不是按业务页面写死的，而是做成了协议驱动的通用层。它接收的是统一 Schema、物料注册表和运行时上下文，自己只负责解析节点、遍历结构、计算变量和事件，再根据节点类型去注册中心找到对应物料进行渲染。这样 Renderer 不需要知道每个业务组件的具体实现细节。新增物料时，只要按协议注册组件定义、默认配置这些信息就能接入；换页面类型时，更多是扩展容器规则或运行时上下文，核心渲染流程还是复用的，所以不需要重写 Renderer。简单说就是 Schema 负责描述，物料负责实现，Renderer 负责解释执行。”

这题想拿高分，记住这 3 句

  • Renderer 不是写死业务组件，而是协议驱动的执行层。
  • 新增物料靠注册，不靠改内核。
  • 页面类型变化更多落在容器和上下文扩展，不应该重写渲染流程。

为什么普通的拖拽命中方案在复杂嵌套布局下不够用？原来的问题具体是什么？ ​

普通拖拽命中方案一般是根据鼠标当前位置去取当前命中的 DOM，或者做简单的矩形碰撞判断。这种方案在结构比较平的页面里问题不大，但在低代码场景下页面往往是多层嵌套的，比如卡片里套分栏、分栏里套表单、表单里再套输入组件。这个时候鼠标实际命中的往往是最内层的叶子节点，而用户想操作的其实是某个外层容器，所以很容易误判。再加上不同容器还有 before、after、inside 三种插入语义，单纯靠当前命中元素很难准确判断最终应该放到哪个父容器、哪个位置。用户侧的直接感受就是组件经常放不进预期位置，或者插入层级错了，拖拽体验很不稳定。所以我们后面才引入了射线检测和最近父容器回溯，去做更稳定的目标容器判断。

你说的“射线检测法 + 最近父容器回溯算法”具体是怎么做的？不用讲代码，重点讲判断流程。 ​

射线检测法是用来解决“我当前鼠标下方是哪个组件？”这一问题的。想象从屏幕里的鼠标指针位置，垂直向画布深处发射一条无形的射线，这条线会穿透一层层的组件。射线首先击中的是离屏幕最近的元素（即 z-index 最高或嵌套最深的那个）。

当通过射线检测找到一个具体的 DOM 节点后，这个节点很可能是一个按钮里的文字，或者一个输入框的内部元素。我们需要找到“我应该把拖拽项放进哪个容器里”。从射线击中的那个具体 DOM 节点开始，沿着 DOM 树向父级、祖父级一层层往上找。在向上回溯的过程中，遇到的第一个符合容器条件的组件实例。

AI回答： “拖拽时我们会先根据鼠标当前位置做一次命中检测，拿到当前指针下最先命中的可见元素。这个元素很多时候不是最终放置目标，可能只是一个文本节点、按钮内部元素或者输入框壳子，所以我们不会直接拿它做插入目标。接下来会先把这个 DOM 节点映射回低代码里的节点实例，再沿着父级链一路往上回溯，找到最近一个满足放置条件的容器节点。这里的‘满足条件’不只是 DOM 层级对，还包括它本身是可放置容器、并且允许接收当前拖拽组件类型。找到目标容器后，再结合鼠标相对容器的位置、容器布局方向和边界阈值，判断当前插入语义是 before、after 还是 inside。这样就不会因为命中了最内层叶子节点，导致组件被错误插到不该去的位置。”

你们是怎么判断 before / after / inside 的？如果是横向布局、纵向布局或者空容器，规则有什么不同？ ​

“插入位置我们是根据鼠标相对目标容器的位置来判断的，但底层语义统一成 before / after / inside 三种。 如果容器是纵向布局，就看鼠标在高度方向上的位置，靠近顶部阈值判成 before，靠近底部阈值判成 after，中间区域判成 inside。 如果容器是横向布局，规则一样，只是把判断轴换成宽度方向，左侧本质上等价于 before，右侧等价于 after。 如果是空容器，因为没有已有子节点做参考，通常直接判成 inside。 另外我们不会用特别生硬的中心线切分，而是会设边界阈值和最小命中区间，避免鼠标在边缘附近移动时插入提示来回抖动，这样用户体验会稳定很多。”

拖拽性能做到 60 FPS，你们具体做了哪些优化？哪些必须实时算，哪些延后算？ ​

“拖拽做到 60 FPS，我们核心是把拖拽链路拆成‘必须实时’和‘可以延后’两部分。 必须实时的只有几件事：当前鼠标位置、目标容器命中、插入位置计算，以及拖拽预览线/占位符的更新。为了让这部分足够轻，我们会在拖拽开始前预计算一份节点和容器的几何信息，拖拽过程中尽量只读缓存，不反复查 DOM；同时把高频 mousemove 的处理放进 requestAnimationFrame，一帧内最多算一次，避免事件频率高于渲染频率。视觉更新上尽量用 transform 和独立拖拽层，减少重排重绘，也避免让整个编辑器树跟着频繁 rerender。 可以延后的部分，比如 Schema 持久化、历史记录入栈、埋点上报、复杂校验、全量布局重算，这些都不放在拖拽主链路里，而是在拖拽结束后或者空闲时再处理。 另外我们还做了命中范围收敛和局部更新，比如只对当前可见区域和候选容器做判断，不让一次拖拽影响整棵页面树。这样下来，拖拽主链路足够轻，复杂页面下帧率才能稳定住。”

一句关键修正 以后这题里把 防抖 换成 requestAnimationFrame 合帧，分数会明显更高。

哪些必须实时，哪些可以延后

必须实时：命中检测、before/after/inside 判断、拖拽预览更新 可以延后：Schema 落库、历史记录、日志埋点、复杂校验、全量持久化

“拖拽触发自动滚动时，最大的风险是 mousemove 和 scroll 两条高频链路叠加，导致布局计算和渲染次数暴增。我们的处理方式是把它们统一收敛到一个 requestAnimationFrame 调度里，每帧只消费一次最新鼠标位置和滚动状态。自动滚动只在顶部/底部边缘阈值区触发，并根据距离边缘的远近动态调整滚动速度。布局上不会在每次滚动时全量重新查 DOM，而是尽量复用拖拽开始时缓存的节点几何信息，通过 scroll delta 修正候选区域，只对当前可见范围和附近容器做命中判断。UI 上也只更新拖拽预览层、目标高亮和插入线，不让整棵画布重渲染。这样即使边滚边拖，主链路也能保持比较轻，帧率会稳定很多。”

滚动导致之前缓存的 rect 失效，你们具体怎么更新缓存？ ​

“我们不会在滚动时每次都重新测量所有 DOM，而是优先利用滚动偏移量 scroll delta 去增量修正缓存的几何信息。比如纵向滚动时，主要更新候选节点和可见容器的 top/bottom，如果存在横向滚动，也会同步修正 left/right。这样拖拽命中还是基于缓存数据做，不需要每一帧都重新查 DOM。只有在发生真正的布局变化，比如节点展开收起、异步内容插入、容器尺寸变化时，才会触发局部重新测量。这样可以把滚动带来的额外开销控制住，保证边滚边拖时性能还是稳定的。”

这句里最关键的一句话你可以记住：

“连续滚动优先做缓存坐标的增量修正，只有布局变化时才重新测量。”

变量编辑器为什么要基于 CodeMirror 6 做，而不是简单 textarea？你们的表达式沙箱具体限制了什么？ ​

CodeMirror 6提供了语法高亮、自动补全、错误提示等功能，这些功能可以提升用户编辑表达式的效率和准确性。用 textarea 实现这些功能比较复杂，需要自己实现语法解析、词法分析、语法树构建等功能，较为复杂。表达式沙箱第一个作用是隔离作用域，防止用户代码访问变量或篡改宿主环境的全局对象（如window、document），限制用户代码对黑名单对象（如DOM操作、网络请求）的访问，第二个作用是保护宿主环境，避免用户代码对全局变量或原型链的意外修改，防止恶意代码直接调用主环境的敏感API。

AI回答： “我们之所以用 CodeMirror 6，核心不是为了把输入框做得更像代码编辑器，而是因为变量表达式本身就不是普通文本输入。低代码场景里的表达式通常会引用页面变量、组件上下文、接口数据和函数能力，如果只是用 textarea，用户虽然也能输入，但缺少语法高亮、上下文补全、错误提示和位置感知，表达式一复杂，编辑体验和出错率都会很差。 CodeMirror 6 比较适合这个场景，一方面它本身比较轻，适合嵌入到低代码平台里做表达式编辑；另一方面它的扩展机制比较灵活，我们可以基于当前页面上下文做变量补全、语法提示、错误标记和自定义规则，而不需要自己从零实现一整套编辑器能力。对低代码来说，这些能力很重要，因为运营或低代码用户不是专业开发者，表达式一旦写错，如果没有及时提示，排查成本会很高。 至于运行时这块，我们又单独做了一层表达式沙箱。它的目标不是宣称绝对安全，而是限制表达式访问宿主环境的能力，比如避免直接访问 window、document 或一些危险 API，同时把表达式执行上下文限制在平台允许暴露的数据和方法里，尽量降低副作用和运行风险。这样编辑体验和运行控制两边才能都兜住。”

为什么是 CodeMirror 6 而不是别的编辑器 不一定要提 Monaco，但如果能补一句会更稳：

你说用“黑名单对象 + Proxy”做表达式沙箱，这种方案的边界在哪里？为什么它不能等同于真正的安全沙箱？ ​

“这个方案的边界很明确，它更像一层运行时风险控制，而不是真正意义上的安全沙箱。因为黑名单思路本质上是 denylist，只能拦截我们已经识别到的危险对象和 API，如果有遗漏，理论上就可能被绕过。Proxy 也只是对暴露出去的对象做访问拦截，它能限制属性读取、函数调用这些行为，但前提是这些对象本身在我们的可控范围内。 更根本的问题是，表达式执行仍然发生在同一个 JavaScript 运行时环境里，并没有像 iframe、Worker 或更严格的隔离机制那样建立真正独立的执行上下文，所以它不能等同于真正的安全沙箱。比如原型链访问、构造器逃逸、间接拿到宿主引用、或者通过我们暴露出去的函数产生副作用，这些都属于它的边界。 所以我们当时对这套方案的定位不是‘绝对安全’，而是‘限制危险能力访问、降低运行风险、减少误操作对宿主环境的影响’。真正敏感的逻辑和安全边界，还是放在后端或者更强隔离机制里处理。”

一句面试加分句 我们做的是前端侧的受控执行环境，不是安全意义上的强隔离沙箱。

那你们最终暴露给表达式的上下文里，到底允许访问哪些东西？你们是怎么做“白名单能力设计”的？ ​

“我们对白名单能力的设计原则是最小暴露、默认只读、按场景分层。表达式里允许访问的主要有几类：第一类是页面级和全局变量，比如页面参数、主题、登录用户这类上下文；第二类是当前组件相关的只读上下文，比如组件 props、绑定值、当前节点 metadata；第三类是已经通过平台拿到的数据快照，比如接口返回结果或数据源结果，但不会让表达式直接发请求；第四类是平台封装过的纯函数能力，比如字符串处理、日期格式化、数组转换这类无副作用工具函数。 我们不会直接把 window、document、fetch、localStorage、组件实例、原始宿主对象这些暴露出去，也不会默认给表达式修改全局状态的能力。也就是说，表达式更像是在一个受控、只读为主的上下文里做计算，而不是在页面里随意执行脚本。”

你们说做了表达式调试器和三层调试架构。表达式执行出错时，用户和开发者分别能看到什么？你们是怎么定位问题的？ ​

“我们当时把表达式调试能力分成三层，并通过权限控制给不同角色看不同信息。 第一层是业务层，面向业务人员，只展示友好的异常提示和受影响的组件状态，比如某个动态文案没渲染出来、某个按钮条件失效，但不会直接暴露技术栈信息。 第二层是实施/配置层，面向搭建人员，除了能看到具体是哪条表达式报错，还能看到当前节点、表达式入参、变量上下文、执行结果和最近一次快照，方便判断是配置问题、变量为空，还是数据结构不符合预期。 第三层是技术层，面向开发者，会进一步展示完整错误堆栈、代码定位、节点元信息、表达式源码、执行耗时，以及前后两次数据快照差异，必要时还能结合事件追踪和性能面板定位到底是表达式逻辑问题、数据注入问题，还是渲染链路问题。

定位流程上，一般是先确认报错节点和表达式，再看执行时的上下文快照和数据差异，判断是表达式写错、变量缺失还是数据结构变化；如果还是不清楚，再下钻到技术层看具体堆栈和源码位置。这样业务、实施和开发三类角色都能在各自可理解的层级里排查问题，不会所有问题都堆给开发。”

你们说“低代码片段插槽机制覆盖了 90% 以上定制化需求”，这个片段插槽到底是什么？为什么它不会把低代码平台重新做成一堆散乱的自定义代码？ ​

“我们说的片段插槽机制，不是让用户在页面里随意写一段脚本，而是在标准物料之外，给平台预留一些受控的扩展点，允许插入局部的自定义渲染或逻辑片段，去覆盖标准能力暂时满足不了的场景。比如一个标准活动组件已经有固定结构，但业务可能只想定制其中某个角标区、按钮区或者状态展示逻辑，这时候没必要为了一点差异重新开发整套物料，就可以通过片段插槽去扩展。

它之所以不会把低代码平台重新做成一堆散乱代码，核心是因为我们做了边界控制。第一，片段只能挂在平台定义好的插槽位上，不允许任意侵入整页；第二，输入输出是受协议约束的，只能拿到指定上下文，比如组件数据、页面变量和平台白名单能力；第三，运行时在受控环境里执行，限制危险 API 和副作用；第四，这些片段本身也能沉淀成可复用资产，而不是每次都零散写一份。 所以它本质上是‘受控扩展’，不是‘放开写代码’，这样才能在保留平台标准化的同时，覆盖高频定制需求。”

一句很加分的话 片段插槽解决的是“局部差异化”，不是“整页自由编程”。

既然有片段插槽，为什么不直接让用户写自定义组件？两者边界怎么划分？ ​

你可以把边界讲成这 4 个维度：

影响范围 片段插槽只解决标准物料里的局部差异，比如某个角标区、文案区、按钮区的展示或一小段交互逻辑；自定义组件通常是一整块完整能力。

复用性 如果只是某个页面、某个业务的局部特殊逻辑，优先用片段插槽；如果这个能力会反复复用，且输入输出稳定，就应该沉淀成自定义组件。

能力复杂度 片段插槽适合轻逻辑、轻渲染的补充；一旦涉及完整样式体系、事件编排、状态管理、设置面板、数据源接入，就更适合做组件。

治理成本 片段插槽是受控扩展，边界小、接入快；自定义组件更强，但开发、测试、文档、配置项设计和后续维护成本都更高，不能把所有差异化都推成组件。

你这题更高分的口语版可以这样说：

“我们不会让所有定制需求都走自定义组件，因为两者解决的问题不一样。片段插槽更适合标准物料里的局部差异化场景，比如某个区域的局部渲染、轻量逻辑或展示规则调整，它的特点是范围小、接入快、边界清晰。自定义组件则更适合沉淀完整能力，当一个需求不仅要改局部展示，还涉及独立样式、事件、状态管理、数据源接入，或者会在多个页面反复复用时，我们才会把它抽成组件。 所以我们的原则是：局部差异优先用片段插槽，完整能力且可复用的需求再升级成自定义组件。这样既能保留平台的灵活性，也不会因为过度开放把低代码平台重新做成一堆散乱的业务代码。”

一句高分总结 拖拽最难，不是因为逻辑最多，而是因为它把计算密集和交互实时性绑在了一起。

“如果现在让我重新做这个低代码平台，我会更早把‘编辑态状态’和‘运行态状态’彻底拆开，并且引入统一的操作层（operation/command layer）去驱动 Schema 变更。 当时为了尽快把平台跑起来，很多交互链路，比如拖拽、属性修改、变量联动，虽然最终都落到 Schema 上，但中间有一部分编辑器临时状态、渲染状态和持久化状态耦合得还比较近。项目初期这样做开发效率高，但平台能力一多，像撤销重做、历史 diff、调试回放、多人协作、性能优化这些需求都会开始互相影响，因为大家都在不同位置直接改状态，边界会慢慢变得不够清晰。

如果重新做，我会更早把状态分成三层：第一层是持久化的 Schema 状态，第二层是编辑器态，比如选中、高亮、拖拽占位、辅助线这些临时 UI 状态，第三层是运行态，比如变量求值结果、数据源结果、表达式执行上下文。所有真正会改页面内容的动作，都统一走 command 或 operation 层，先描述‘发生了什么变更’，再决定怎么更新 Schema、怎么记录历史、怎么触发渲染。 这样做的好处是很明显的：第一，撤销重做、历史回放和 diff 能天然建立在操作流上；第二，插件边界会更清楚，扩展能力不用直接侵入核心状态；第三，像拖拽这种高频交互可以更多停留在编辑器态，不用每一帧都去污染持久化数据，对性能也更友好；第四，如果未来要做多人协作或者更细粒度的调试追踪，这套架构也更容易演进。

所以如果说只选一个我会重做的架构点，我会选‘状态分层 + 统一操作层’。因为它不是某个单点优化，而是会直接影响平台后续的可扩展性、可维护性和性能上限。”

“如果重做，我会更早把编辑器态和运行态从 Schema 里拆开，并引入统一的 command/operation 层。因为当时为了加快落地，拖拽、属性修改、变量联动这些链路有些地方还是直接改状态，前期开发快，但后面做撤销重做、历史 diff、调试回放和性能优化时，状态边界会不够清晰。 如果重来，我会把状态拆成持久化 Schema、编辑器临时状态、运行时状态三层，所有真正修改页面内容的动作统一走操作层。这样历史记录、插件扩展、性能优化，甚至未来多人协作都会更容易演进。”

你也可以记一个特别好用的总结句：

如果重做，我最想提前做好的不是某个功能，而是状态边界和变更路径。

事件编排：事件编排本质上是把页面里的交互流程做成可配置能力。它通常由触发器、条件判断和动作链组成，比如点击按钮后先校验表单，再请求接口，再更新变量、弹提示或跳转页面。这样很多常见业务交互就不需要开发手写逻辑了。对低代码平台来说，事件编排是把平台从‘静态搭页面’提升到‘可配置业务流程’的关键能力之一。

“低代码平台里的事件编排引擎，本质上是一个配置驱动的流程执行器。它一般会包含触发层、流程协议层、节点执行层和运行时上下文层。触发层负责定义什么时候启动流程，协议层负责把动作链描述成结构化配置，执行层按节点顺序调度接口请求、变量更新、条件判断、弹窗、跳转这些动作，上下文层负责在节点之间传递变量、接口结果和页面状态。为了真正可用，通常还要补上调试和容错能力，比如节点日志、输入输出快照、错误定位和失败分支处理。这样平台才能把很多业务交互从手写代码变成可配置流程。”

requestAnimationFrame 合帧：把很多次高频触发的计算，合并到浏览器下一帧渲染前只执行一次。

requestAnimationFrame 合帧的意思是，把高频事件产生的多次更新请求，收敛到浏览器下一帧绘制前统一执行一次。比如拖拽场景下，mousemove 可能一帧内触发很多次，但真正渲染只发生一次，所以我们会先记录最新鼠标位置，再在 requestAnimationFrame 回调里统一做命中检测和预览更新，这样既能减少重复计算，又能保证交互足够实时。”

拖拽里哪些逻辑应该放到 requestAnimationFrame，哪些不应该放进去？

“拖拽里适合放进 requestAnimationFrame 的，一般都是和当前帧视觉反馈直接相关的逻辑，比如拖拽预览位置、目标容器命中、before / after / inside 判断、自动滚动决策，以及高亮容器和插入线这些局部 UI 更新。因为这些必须跟着鼠标实时变化。 而像 Schema 持久化、历史记录、埋点、复杂校验这类不影响当前帧视觉结果的逻辑，就不应该放进 requestAnimationFrame，否则会把主链路拖重，影响帧率。我们的原则是让 requestAnimationFrame 只承担这一帧必须完成的最小工作集。”

那拖拽里怎么避免在 requestAnimationFrame 里产生强制同步布局（forced reflow）？

“为了避免在 requestAnimationFrame 里产生强制同步布局，我们会尽量把几何信息前置缓存，拖拽过程中只读缓存；如果一帧里既要读又要写，也会先统一读取，再统一更新 UI，避免写完样式立刻读 rect。视觉更新主要走独立拖拽层和 transform，滚动时优先用 scroll delta 修正缓存，只有发生真实布局变化时才局部重新测量 DOM。这样可以尽量避免 forced reflow，把拖拽主链路控制得更轻。”

拖拽开始前通常要预计算哪些缓存数据？

“拖拽开始前，我们会先预计算一批缓存数据，包括候选节点的几何信息、节点层级关系、容器可放置元信息、DOM 到 Schema 节点的映射、滚动容器状态以及当前拖拽对象自身信息。这样拖拽过程中每一帧只需要基于缓存做命中、回溯和插入位置判断，不需要反复查 DOM 或扫描整棵树，性能会稳定很多。”

这些缓存什么时候失效，什么时候必须重新测量？

“缓存并不是一变化就全量失效，我们会先区分是位置变化还是布局变化。像滚动这种只是视口偏移变化的场景，通常通过 scroll delta 去增量修正缓存坐标就够了，不需要重新测量 DOM。只有在节点展开收起、异步插入删除、容器尺寸变化、布局方向变化这类真正影响几何关系的场景下，才会触发重新测量。重新测量也尽量优先做局部，比如只重测受影响的子树或候选区域，只有全局布局变化时才重建整份缓存。这样既保证命中准确性，也能把拖拽性能控制住。”

“为什么不用 document.elementFromPoint 直接做命中，就完了？”

“document.elementFromPoint 我们可以用，但更多是把它当作命中检测的入口，而不是最终放置结果。因为它只能告诉我当前像素点上最上层的 DOM 是谁，但低代码拖拽真正关心的是这个 DOM 对应哪个节点、它是不是合法容器、当前拖拽组件能不能放进去，以及插入语义是 before / after / inside。另外拖拽过程中还会有预览层、辅助线这些浮层干扰，所以不能直接拿 elementFromPoint 的结果当最终目标。更合理的做法是先命中，再映射节点、回溯父容器、最后结合布局规则做插入判断。”

那为什么不用现成拖拽库直接解决这些问题，要自己做命中和编排算法？

核心结论可以先记住：

  • 现成拖拽库能解决“拖得动”，但不一定能解决“低代码里复杂嵌套场景下该放到哪、怎么放、放完后怎么更新协议”。

也就是说：

  • 拖拽库擅长处理的是通用交互层，比如拖拽开始、移动、结束、缩放、基础碰撞
  • 你们真正自研的是低代码语义层，比如容器命中、父容器回溯、before/after/inside、Schema 更新、占位符和辅助线逻辑

我们没有完全从零手写所有拖拽交互，而是分层处理。底层像拖拽手势、基础移动、缩放这些通用交互能力，会借助 interactjs 这类库去提升开发效率；但低代码编辑器真正难的部分不在‘能不能拖动’，而在‘复杂嵌套结构下该放到哪、能不能放、放进去后 Schema 怎么更新’。这些能力和我们自己的节点模型、容器规则、渲染协议强相关，现成库没法直接给，所以像射线命中、父容器回溯、before / after / inside 判断、占位符和辅助线逻辑、以及拖拽后的协议更新，这部分我们是自己实现的。 所以更准确地说，我们不是完全自研拖拽，而是底层交互复用成熟库，上层编排和放置算法按低代码场景自研。”

那你们最终是怎么把一次拖拽结果落到 Schema 上的？

“我们不会在拖拽过程中频繁直接改 DOM 或持久化 Schema，而是先把拖拽过程里的命中结果维护成临时放置信息，比如目标容器、参考节点和 before / after / inside 语义。等用户真正 drop 的时候，再把这次拖拽转换成一次结构操作。 如果是已有节点移动，就先从原父节点里移除，再按最终位置插入目标父节点；如果是新物料，就先生成节点和默认配置，再插入到目标位置。因为我们的 Schema 把结构关系和节点详情拆开了，所以这个阶段主要改的是父子关系和节点顺序，必要时补一份节点详情。最后再由 Renderer 根据新的 Schema 做局部更新，同时记录操作历史，用于撤销重做。 本质上，拖拽改变的不是 DOM，而是节点关系，DOM 只是 Schema 变化后的渲染结果。”

那撤销重做（undo/redo）你们是怎么设计的？是存整份 Schema 快照，还是存操作记录？ ​

“撤销重做我们更倾向于以操作记录为主，而不是每次都存整份 Schema。因为低代码平台操作非常频繁，如果每次属性修改、拖拽、样式调整都保存完整快照，内存和性能压力会比较大，而且也不利于理解用户做了什么。 所以我们会把编辑行为抽象成有语义的 operation，比如插入节点、移动节点、删除节点、修改属性。以拖拽为例，一次操作只需要记录节点从哪个父容器、哪个索引，移动到了哪里，undo 时执行逆操作就可以了。 当然纯操作流历史太长也会有成本，所以工程上一般会配合 checkpoint 或 snapshot，用‘操作记录为主、快照兜底’的方式做恢复。这样既保留了操作语义，又能把恢复成本控制住。”

那属性连续输入这种高频操作，历史记录怎么合并？不然用户按一下退格都要 undo 一次。

“历史记录我们不会机械地一变就入栈，而是会做操作合并。像文本输入、滑块调整、颜色拖选这类高频连续操作，如果每一步都单独记录，用户按一次 undo 只能回退一个字符或者一个像素，体验会很差，所以这类操作通常会按时间窗口、操作类型和目标字段进行合并。比如同一个节点的同一个属性，在连续输入期间都归为同一条历史记录，停顿、失焦或切换目标时再真正提交。 而像新增节点、删除节点、复制、一次完整拖拽结束这类语义明确的动作，就会单独入栈，不做合并。我们的原则是：undo/redo 要匹配用户对‘一次操作’的心理预期，而不是机械地对应每一次状态变化。”

那如果是拖拽过程中实时更新画布，但又只想在 drop 时入栈，状态和历史怎么协调？

“我们会把拖拽过程中的实时反馈和最终持久化操作拆开处理。拖拽中更新的主要是编辑器临时状态，比如当前命中容器、占位符位置、插入语义和拖拽预览层，这些状态只服务交互反馈，不会每一帧都去改 Schema 或写历史。等用户真正 drop 的时候，再根据最终目标位置生成一次正式 operation，比如节点移动或新物料插入，统一应用到 Schema 并入历史栈。这样既能保证拖拽过程足够轻量，也能让 undo/redo 的粒度符合用户对‘一次拖拽’的理解。”

那如果拖拽过程中，画布为了预览效果已经临时重排了，drop 失败或取消时怎么回滚？

“我们会尽量把拖拽过程中的预览重排控制在临时层，而不是直接改正式 Schema。比较轻的方式是只用占位符、插入线和高亮来模拟结构变化；如果需要更真实的预览，也是在内存里维护一份影子结构去驱动画布显示。这样当用户取消拖拽或者 drop 失败时，不需要逐步反向回滚中间过程，只要直接丢弃这份预览态，恢复到 dragStart 时的 committed state 就可以了。核心原则是 dragging 期间只做预演，drop 成功后才正式提交。”

取消拖拽时不是回放逆操作，而是直接丢弃预览态，恢复 committed state。”

你觉得这个项目里最体现你架构能力的一个设计决策是什么？ ​

“如果只选一个最体现我架构能力的设计决策，我会选‘把平台核心建立在统一 Schema 协议之上，并把 Renderer、物料和编辑器解耦’。 因为低代码平台最容易失控的地方在于，编辑器、渲染层、物料系统和发布链路都在围绕页面配置工作，但如果没有一套稳定的数据协议，大家就会各自维护自己的页面结构和运行逻辑。短期看开发快，长期会非常难扩展，尤其是一旦物料变多、页面类型变多、调试和版本迁移能力跟进时，系统复杂度会迅速上升。

所以我们当时没有把页面简单存成一棵普通组件树，而是专门设计了 Schema 协议，把结构关系、节点详情、样式、事件、变量绑定这些信息做了明确分层。基于这套协议，编辑器只负责生产和修改 Schema，Renderer 只负责解释协议并渲染，物料只负责按约定提供元数据和实现。这样带来的价值很明显：新增物料时更多是注册而不是改渲染内核；做版本迁移、调试工具、历史 diff、数据校验时，也都能围绕统一协议展开；平台后面扩展不同页面类型时，复用的也是同一套核心执行链路，而不是推倒重写。

我觉得这个决策最重要的地方在于，它不是解决某一个局部功能，而是把整个平台从‘功能集合’变成了‘协议驱动系统’。这会直接决定平台后续的扩展性、维护成本和演进上限。”

1 分钟口语版 “如果只选一个最体现架构能力的点，我会选 Schema 协议驱动这件事。因为低代码平台里，编辑器、Renderer、物料、发布和调试其实都围绕页面配置在工作，如果没有统一协议，后面一定会越来越耦合。我们当时专门设计了 Schema，把结构关系、节点详情、样式、事件、变量这些信息分层定义清楚，然后让编辑器负责产出 Schema，Renderer 负责解释执行，物料负责按协议注册实现。这样新增物料、做版本迁移、做调试工具、扩展页面类型时，都不用频繁改核心链路。我觉得这不是某个单点优化，而是决定平台能不能长期演进的架构基础。”

你也可以记住这句总结 真正的架构能力，不是把功能做多，而是先定义清楚平台内部的通用语言和边界。
