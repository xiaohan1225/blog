你刚才说“自定义性能指标”是促使你们自研的重要原因。那你详细讲一下，你们平台里的“秒开率”是怎么定义和计算的？ ​

秒开率，顾名思义就是1s成功打开页面首屏的用户占比，这是衡量系统性能的重要指标，在用户加载页面的时候会通过MutationObserver进行采集，我们设计了一套首屏检测算法来进行检测。首屏时间的标准，在不同环境下有一定差异，SSR 环境是秒开率达到80%以上，端内（Hybrid 环境）是 65%以上，端外（浏览器环境）是 40%以上。另外，还有1.5秒开率，2秒开率等指标，也分别有不同标准。FCP/LCP作为性能的标准指标，可以衡量首次内容渲染和最大内容渲染，但是实际情况中往往和用户的实际感受有所差异，比如首屏只有一张图片，但图片并没有加载出来，此时 FCP/LCP 已经结束，然而用户还是没看到首屏内容。

AI回答： “我们定义的秒开率，本质上是首屏在 1 秒内首屏成功打开的用户占比。 它不是简单看某个性能 API 的单点时间，而是更强调用户视角下‘页面是不是已经真的打开了’。 采集上我们基于 MutationObserver 监听首屏区域 DOM 结构变化，结合元素可见性、资源加载状态、接口数据驱动渲染等信号，计算首屏完成时间，再统计 1 秒内完成的占比。 之所以这么做，是因为 FCP/LCP 只能反映浏览器渲染阶段的关键节点，但不一定等于用户真正看到了可交互、有业务价值的内容。比如首屏主要内容依赖接口返回，或者是图片、视频资源，浏览器可能已经产生了首次绘制，但用户仍然感知为空白或不可用。 阈值上，我们按场景分层定义目标，比如 SSR、Hybrid、普通浏览器环境分别设置不同基线，因为它们在容器能力、网络条件和渲染路径上差异很大。这个指标比单纯看 FCP/LCP 更适合我们，因为它能更真实反映业务页面体验，也更方便和转化、跳出率做关联分析。””

你详细讲一下基于 MutationObserver 的首屏检测算法是怎么做的？ ​

它能监听 DOM 树的更改并执行相关的回调。核心的统计思路就是：在页面初始化时，使用 MutationObserver 监听 DOM 元素，当其发生变化时，程序会标记变化的元素，并记录时间点和分数，存储到数组中，当达到如下条件时，说明首屏渲染已经结束：

  • 计算时间超过 30s 还没结束。
  • 计算了 4 次且 1s 内分数不再变化。
  • 计算了 9 次且分数不再变化。

统计分数过程如下：

  • 递归遍历 DOM 元素及其子元素，根据元素层级设定元素权重。层级越深的元素最接近用户看到的内容，权重也就越高。比如第一层权重为 1 ，渲染完成得 1 分，每增加一层权重增加 0.5，第三层的权重为 3.5，也就是渲染完成得 3.5 分。

最终，我们拿到一个记录了时间点和分数的数组，然后通过数组的后一项 - 数组前一项求出元素分数变化率，找到变化率最大点的分数对应的时间，即为首屏时间。

AI回答： 我们不是简单把 DOM 变化时间当成首屏时间，而是把首屏渲染过程看成一个持续增量完成的过程。 页面初始化后，通过 MutationObserver 监听首屏区域内 DOM 的新增和变化，每次变更后递归计算可见节点的得分。 这个得分不是纯节点数，而是结合节点层级、可见性、是否在首屏视口内、是否属于真实内容节点来综合计算。因为真正被用户感知到的首屏，通常是内容结构逐步稳定的过程。 随着页面渲染推进，得分会持续增长；当得分在一段时间内趋于稳定，或者达到某个最大变化拐点后不再明显增长，我们就认为首屏核心内容已经完成。

如果页面里有下面这些场景，你的算法怎么避免误判？

  • 骨架屏先渲染出来了，但真实内容还没回来

  • 首屏核心区域是一张大图，DOM 早就有了，但图片资源还没加载完成

  • SSR 页面 HTML 已经输出了，但客户端 hydration 还没完成

  • 接口是分批返回的，页面会连续渲染多次

  • 针对骨架屏，我们可以识别到骨架屏之后进行排除，等待真实DOM加载完成后再进行首屏检测。

  • 需要统计图片加载时间，拿之前计算的首屏时间进行对比，取最大值作为首屏时间

  • SSR页面虽然已经输出了 HTML，但我们计算的是真实 DOM，所以没关系。

  • 对于分批返回的接口，我们会等待页面稳定后再作为首屏时间。

  • 另外还有两种情况，比如首屏还未统计完用户进行了滚动，导致首屏时间被拉长，还有用户点击跳走了页面，而首屏时间统计脚本还在执行，会导致统计到错误的页面。针对第一种情况，在用户进行操作后，我们会放弃这次的首屏时间统计，针对第二种情况，我们在统计前会先确定是哪个页面进行统计，而不是统计完后再判断。

AI回答： “骨架屏我们一般不直接计入有效首屏内容，会通过节点特征和业务约定进行过滤，比如统一骨架容器 class、占位元素特征、重复块结构等；只有真实数据节点出现后才开始计入分数。 图片场景下，DOM 出现不代表用户真的看到了内容，所以如果首屏关键节点是图片、视频这类资源型元素，会继续结合资源加载完成事件，最终首屏时间取 DOM 完成时间和关键资源完成时间的较大值。 SSR 场景下，服务端输出 HTML 只能说明有静态内容，不代表页面真正完成可用，所以我们会区分首屏可见时间和 hydration 完成后的稳定时间，避免把未完成接管的页面过早算作完成。 接口分批返回场景下，我们不会按第一次渲染就结束，而是看有效内容得分是否进入稳定期；如果后续还有连续增量渲染，就延后首屏完成时刻，直到页面趋于稳定。”

hydration 完成后的稳定时间： “我们不会直接把 SSR 输出 HTML 的时间当成首屏完成，而是会在客户端接管后，持续观察首屏区域内的有效节点变化。只有当首屏内容节点在一个稳定窗口内不再增长，关键图片和接口也完成，且没有新的 hydration 替换行为时，才把这个时间点认定为 hydration 完成后的稳定时间。”

满足以下条件时，判定为稳定：

  1. 首屏区域连续 500ms 无有效 DOM 新增/替换
  2. 首屏关键图片资源已完成加载
  3. 首屏关键接口请求已结束
  4. 页面未发生路由切换、滚动打断、卸载

当时线上出现某个问题触发邮件预警，由于是一个大范围的持续性问题，所以导致预警邮件越来越多，写数据（产生告警日志）的速度远大于读数据（展示、处理告警）的速度，且后台处理逻辑存在瓶颈，所以导致数据看不全。

针对这个问题，我们采用了以下方法：

  1. 引入“告警抑制”和“降噪”机制，让系统报警更加智能。第一个是状态翻转机制（Flapping），之前每次检测到性能超标就发邮件，问题恢复又发邮件，反复波动，现在改成只有当问题持续一定时间（例如持续5分钟）或连续发生N次探测失败后，才触发警报。问题恢复后，按告警级别设置不同重发窗口。（比如P2/P3级别的日志24 小时内不再重复发送相同原因邮件）第二个是沉默周期/告警聚合。之前是某个 JS 错误短时间爆发，会发多封邮件，现在改为将同一时间段内（如30分钟内）的同类错误合并为一封邮件，标题注明“有N个接口发生超时”，而非逐一发送。
  2. 实施告警分级与分渠道处理，只在前端展示紧急的实时数据，历史数据通过“查询”功能获取。
  • P0/P1（紧急）： 短信/电话，立即处理。
  • P2/P3（警告）： 邮件，但无需立即展示在前端实时看板，可归档。
  • P4（信息）： 仅入库，供排查问题时查询。
  1. 冷热数据分离
  • 分表/分区： 按天甚至按小时分区。查询2小时内的数据，只需要扫描最近的分区，速度会很快。
  • 数据归档： 将超过24小时或48小时的预警数据，自动转移到归档表或归档数据库（甚至是大数据平台）。前端默认只查询热数据，需要看历史数据时，需要加时间筛选并查询慢一些的归档库。
  1. 限流与熔断：为了在极端情况下，需要保护监控系统自身。引入丢弃策略，即如果同一时刻产生大量重复预警（比如机房断电，上万台机器同时报失联），系统应该能够识别并丢弃非核心的告警，只发送一条“大面积服务器失联”的汇总通知。

通过引入这四个策略，有效解决了这个问题。

“这个问题本质上不是单点故障，而是告警系统在高噪声场景下出现了级联拥塞。 一方面，持续性异常导致大量重复告警被不断生成；另一方面，告警计算、入库、查询和通知链路都被高频重复事件挤占，最终实时看板延迟、通知泛滥、处理人也逐渐麻木。 所以我们的治理思路不是单纯减少告警，而是把‘重复、低价值、短时抖动’的噪声从系统里提前拦截掉，只把持续、扩散、影响大的问题升级出来。”

我想听你回答：

INP 本质上衡量的是什么 它和长任务、主线程阻塞之间的关系 如果一个页面 INP 差，你会从哪些维度继续下钻 最后可能落到哪些前端问题上，比如事件处理、渲染、第三方脚本、数据量过大等

INP 指的是页面首次交互（点击链接、按钮等）到浏览器实际响应的延迟，它衡量的是用户交互的流畅度，如果页面中存在长任务导致主线程阻塞，那么就会导致 INP 值增加。如果一个页面 INP 差，我会优化用 PerformanceObserver 监控是否有长任务，如果没有长任务，则需要排查业务代码是否有性能问题，比如某些点击事件需要异步执行后才给用户反馈，又没有给 loading 提示，导致用户体验很差。

“INP 反映的是用户交互到下一次可见更新之间的整体响应质量，它不只看第一次交互，而是关注页面生命周期中较差的那批交互表现。 它通常和主线程阻塞高度相关，因为用户输入、事件处理、样式计算、布局、绘制都要竞争主线程；一旦有长任务，交互就会排队，INP 就容易变差。 如果发现某个页面 INP 很差，我会先按交互链路拆解： 第一，先看是不是输入延迟高，也就是交互发生时主线程是否正被长任务占满； 第二，看事件处理本身是否做了重计算，比如同步循环、复杂 JSON 处理、大量状态更新； 第三，看事件触发后是否带来了重渲染、频繁布局计算或大面积 DOM 更新。 继续下钻时，我会结合 PerformanceObserver 看 long task，结合用户行为埋点看是哪一类交互触发问题，再结合性能面板或自定义埋点去定位具体组件、函数和第三方脚本。”

比如： 容器很早就渲染出来了，但里面还是骨架屏 容器 DOM 已经在了，但关键图片还没加载完 容器是异步分批渲染的，先出来一半内容

骨架屏我们一般不直接计入业务化 FMP，会通过节点特征和业务约定进行过滤，比如统一骨架容器 class、占位元素特征、重复块结构等；只有真实数据节点出现后才开始计入业务 FMP。 图片场景下，DOM 出现不代表用户真的看到了内容，所以如果页面关键节点是图片、视频这类资源型元素，会继续结合资源加载完成事件，最终首业务化 FMP 时间取 DOM 完成时间和关键资源完成时间的较大值。 异步分批渲染场景下，我们不会按第一次渲染就结束，而是看有效内容得分是否进入稳定期；如果后续还有连续增量渲染，就延后业务 FMP完成时刻，直到页面趋于稳定。

AI回答： “我们不会把业务方传入的关键容器一出现就直接当成业务化 FMP 完成，因为容器出现不等于用户真正看到了可用内容。为了避免误判，我们会结合内容类型和页面渲染过程做二次判断。

第一种是骨架屏场景。容器可能很早就渲染出来，但里面还是占位内容，这时候不能计入业务化 FMP。我们会通过统一骨架 class、占位节点特征、重复结构特征或者业务约定标记，把骨架节点过滤掉，只有真实数据节点进入容器后才开始计时或更新完成状态。

第二种是资源型内容场景，比如图片、视频、canvas。DOM 出现不代表内容已经真正可见，所以如果关键区域依赖这类资源，我们会继续监听资源加载完成事件。最终时间不会只看容器挂载时刻，而是取‘容器有效内容出现时间’和‘关键资源加载完成时间’中的较大值。

第三种是异步分批渲染场景。有些业务区域会先渲染一部分，再通过接口逐步补齐。如果第一次渲染就结束，会明显偏早。对于这种情况，我们会持续观察关键容器内有效内容的增量变化，结合 MutationObserver 和稳定窗口判断，只有当内容在一段时间内不再明显增长，或者关键节点都已完成渲染时，才认定业务化 FMP 完成。

第四种是业务数据驱动场景。有些容器虽然 DOM 已经稳定，但真正用户关心的数据还没回来，比如价格、库存、抽奖结果等。对于这类核心场景，我们会支持业务侧传入额外的完成标记，或者结合关键接口结束时间一起判断，避免只从 DOM 角度过早结束。

所以我们最终不是采‘容器出现时间’，而是采‘关键区域进入真实可感知、基本稳定状态的时间’。这样虽然比标准指标复杂一些，但更贴近业务实际体验。”

疑问 ​

插件降级和熔断是什么意思？

插件降级和熔断，都是为了防止“监控 SDK 自己出问题，反过来影响业务页面”。

插件降级 是指： 当某个插件能力异常、性能开销过大、或者当前环境不支持时，主动关闭一部分非核心能力，只保留最基础的采集能力。

比如：

行为轨迹插件太重，先关闭行为回放，只保留关键点击埋点 性能插件在某些老浏览器不支持完整指标时，退化为只采基础时间点 某些增强分析能力先停掉，但错误上报还保留 核心思想是： 功能变少，但系统还能工作

插件熔断 是指： 如果某个插件持续报错、进入死循环、或者短时间内异常次数过多，就直接把这个插件停掉，避免它继续消耗资源或影响业务。

比如：

某个插件初始化后连续抛错 10 次 某个插件导致大量同步计算，明显拖慢页面 某个插件上报失败重试过多，导致内部队列堆积 这时候系统会：

停止执行这个插件后续逻辑 记录一条内部错误日志 必要时上报“插件已熔断”的状态 核心思想是： 先切断故障源，保护整体系统

你在面试里可以这么说：

“插件降级是主动减少能力范围，保证核心采集还能继续；插件熔断是当单个插件持续异常时，直接停止它的执行，避免故障扩散。两者本质上都是为了保证监控 SDK 的问题不会影响业务页面。”

一句话区分：

降级：还能跑，但少跑点 熔断：先别跑了，避免继续出事

全链路 ​

从用户访问页面开始，到页面渲染、用户交互、接口请求、异常发生、告警触发、问题定位这一整条链路的可观测性闭环。

也就是说，它不是只监控某一个点，比如只看 JS 报错，或者只看性能，而是把前端线上问题相关的关键环节都串起来。

你可以把“全链路”拆成这几段来讲：

用户访问链路 用户进入页面、路由跳转、页面曝光、设备/环境信息

页面渲染链路 首屏时间、LCP、INP、白屏、业务关键区域渲染时间、卡顿

用户行为链路 点击、输入、提交、关键业务操作、行为路径

请求链路 API 请求发起、耗时、状态码、超时、失败、重试

异常链路 JS 错误、Promise 错误、资源加载失败、自定义业务异常

排障链路 日志聚合、错误指纹、SourceMap 还原、告警分级、趋势分析、会话回溯

所以“全链路”的重点不是“全站所有系统都监控”，而是：

把前端问题从发生到定位的关键数据链路打通

你在面试里可以这样答：

“这里的全链路，不是指传统后端那种微服务调用链 tracing，而是指前端侧从用户访问、页面渲染、用户行为、接口请求、异常上报到告警分析和问题定位这一整套链路被统一串起来。这样我们不仅知道页面报错了，还能知道用户当时做了什么、请求有没有异常、性能是不是劣化、问题集中在哪个版本，最终形成从发现到定位的完整闭环。”

如果你想更高级一点，可以补一句区分概念：

“它和后端链路追踪不完全一样。后端更强调服务调用链路和 trace，而前端全链路更强调用户体验链路和排障闭环。”

G2/G2Plot 根本画不了这种节点+边的关系结构。

G6 专门为此设计，内置了布局算法（力导、树形、放射状等）、交互（拖拽、高亮邻居节点）和丰富的节点样式。

简单判断： 如果你的数据结构包含 nodes（节点）和 edges（连线），就必须用 G6。
