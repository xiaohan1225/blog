为什么当时要自研这套监控 SDK 和可视化平台，而不是直接接入现成方案，比如 Sentry、ARMS 之类？你们当时具体遇到了什么业务问题，现成方案又为什么不能满足？请你结合“业务诉求、技术限制、成本取舍、最终收益”来回答。 ​

“当时选择自研，核心不是为了重复造轮子，而是因为我们业务对监控的要求已经不只是通用错误采集，而是需要一套贴合内部业务闭环的前端全链路监控能力。 第一，标准方案能覆盖 JS 报错和部分性能指标，但我们更关注像秒开率、白屏、做题提交流程成功率这类强业务属性指标，这些需要结合页面渲染阶段、接口链路和用户行为去定义，通用平台很难直接支持。 第二，我们希望把错误监控、性能监控、用户行为轨迹、SourceMap 解析、告警分级、业务看板放到一个统一平台里，方便研发和业务一起定位问题，而不是多个系统来回切换。 第三，公司有 10+ 核心业务，技术栈和场景不完全一致，我们希望 SDK 足够轻量、可插件化，并且能按内部规范定制采样、上报、脱敏、告警策略，这类深度定制用现成方案改造成本也不低。 第四，商业产品按量计费，在我们百万级日志量场景下长期成本较高，所以综合能力边界和使用成本，最终选择自研。 最终落地后，故障发现从用户反馈变成系统分钟级告警，排障效率和核心页面体验都有明显提升，也证明这套自研方案是值得的。””

你刚才说“自定义性能指标”是促使你们自研的重要原因。那你详细讲一下，你们平台里的“秒开率”是怎么定义和计算的？ ​

秒开率，顾名思义就是1s成功打开页面首屏的用户占比，这是衡量系统性能的重要指标，在用户加载页面的时候会通过MutationObserver进行采集，我们设计了一套首屏检测算法来进行检测。首屏时间的标准，在不同环境下有一定差异，SSR 环境是秒开率达到80%以上，端内（Hybrid 环境）是 65%以上，端外（浏览器环境）是 40%以上。另外，还有1.5秒开率，2秒开率等指标，也分别有不同标准。FCP/LCP作为性能的标准指标，可以衡量首次内容渲染和最大内容渲染，但是实际情况中往往和用户的实际感受有所差异，比如首屏只有一张图片，但图片并没有加载出来，此时 FCP/LCP 已经结束，然而用户还是没看到首屏内容。

AI回答： “我们定义的秒开率，本质上是首屏在 1 秒内首屏成功打开的用户占比。 它不是简单看某个性能 API 的单点时间，而是更强调用户视角下‘页面是不是已经真的打开了’。 采集上我们基于 MutationObserver 监听首屏区域 DOM 结构变化，结合元素可见性、资源加载状态、接口数据驱动渲染等信号，计算首屏完成时间，再统计 1 秒内完成的占比。 之所以这么做，是因为 FCP/LCP 只能反映浏览器渲染阶段的关键节点，但不一定等于用户真正看到了可交互、有业务价值的内容。比如首屏主要内容依赖接口返回，或者是图片、视频资源，浏览器可能已经产生了首次绘制，但用户仍然感知为空白或不可用。 阈值上，我们按场景分层定义目标，比如 SSR、Hybrid、普通浏览器环境分别设置不同基线，因为它们在容器能力、网络条件和渲染路径上差异很大。这个指标比单纯看 FCP/LCP 更适合我们，因为它能更真实反映业务页面体验，也更方便和转化、跳出率做关联分析。””

你详细讲一下基于 MutationObserver 的首屏检测算法是怎么做的？ ​

它能监听 DOM 树的更改并执行相关的回调。核心的统计思路就是：在页面初始化时，使用 MutationObserver 监听 DOM 元素，当其发生变化时，程序会标记变化的元素，并记录时间点和分数，存储到数组中，当达到如下条件时，说明首屏渲染已经结束：

  • 计算时间超过 30s 还没结束。
  • 计算了 4 次且 1s 内分数不再变化。
  • 计算了 9 次且分数不再变化。

统计分数过程如下：

  • 递归遍历 DOM 元素及其子元素，根据元素层级设定元素权重。层级越深的元素最接近用户看到的内容，权重也就越高。比如第一层权重为 1 ，渲染完成得 1 分，每增加一层权重增加 0.5，第三层的权重为 3.5，也就是渲染完成得 3.5 分。

最终，我们拿到一个记录了时间点和分数的数组，然后通过数组的后一项 - 数组前一项求出元素分数变化率，找到变化率最大点的分数对应的时间，即为首屏时间。

AI回答： 我们不是简单把 DOM 变化时间当成首屏时间，而是把首屏渲染过程看成一个持续增量完成的过程。 页面初始化后，通过 MutationObserver 监听首屏区域内 DOM 的新增和变化，每次变更后递归计算可见节点的得分。 这个得分不是纯节点数，而是结合节点层级、可见性、是否在首屏视口内、是否属于真实内容节点来综合计算。因为真正被用户感知到的首屏，通常是内容结构逐步稳定的过程。 随着页面渲染推进，得分会持续增长；当得分在一段时间内趋于稳定，或者达到某个最大变化拐点后不再明显增长，我们就认为首屏核心内容已经完成。

如果页面里有下面这些场景，你的算法怎么避免误判？

  • 骨架屏先渲染出来了，但真实内容还没回来

  • 首屏核心区域是一张大图，DOM 早就有了，但图片资源还没加载完成

  • SSR 页面 HTML 已经输出了，但客户端 hydration 还没完成

  • 接口是分批返回的，页面会连续渲染多次

  • 针对骨架屏，我们可以识别到骨架屏之后进行排除，等待真实DOM加载完成后再进行首屏检测。

  • 需要统计图片加载时间，拿之前计算的首屏时间进行对比，取最大值作为首屏时间

  • SSR页面虽然已经输出了 HTML，但我们计算的是真实 DOM，所以没关系。

  • 对于分批返回的接口，我们会等待页面稳定后再作为首屏时间。

  • 另外还有两种情况，比如首屏还未统计完用户进行了滚动，导致首屏时间被拉长，还有用户点击跳走了页面，而首屏时间统计脚本还在执行，会导致统计到错误的页面。针对第一种情况，在用户进行操作后，我们会放弃这次的首屏时间统计，针对第二种情况，我们在统计前会先确定是哪个页面进行统计，而不是统计完后再判断。

AI回答： “骨架屏我们一般不直接计入有效首屏内容，会通过节点特征和业务约定进行过滤，比如统一骨架容器 class、占位元素特征、重复块结构等；只有真实数据节点出现后才开始计入分数。 图片场景下，DOM 出现不代表用户真的看到了内容，所以如果首屏关键节点是图片、视频这类资源型元素，会继续结合资源加载完成事件，最终首屏时间取 DOM 完成时间和关键资源完成时间的较大值。 SSR 场景下，服务端输出 HTML 只能说明有静态内容，不代表页面真正完成可用，所以我们会区分首屏可见时间和 hydration 完成后的稳定时间，避免把未完成接管的页面过早算作完成。 接口分批返回场景下，我们不会按第一次渲染就结束，而是看有效内容得分是否进入稳定期；如果后续还有连续增量渲染，就延后首屏完成时刻，直到页面趋于稳定。”

hydration 完成后的稳定时间： “我们不会直接把 SSR 输出 HTML 的时间当成首屏完成，而是会在客户端接管后，持续观察首屏区域内的有效节点变化。只有当首屏内容节点在一个稳定窗口内不再增长，关键图片和接口也完成，且没有新的 hydration 替换行为时，才把这个时间点认定为 hydration 完成后的稳定时间。”

满足以下条件时，判定为稳定：

  1. 首屏区域连续 500ms 无有效 DOM 新增/替换
  2. 首屏关键图片资源已完成加载
  3. 首屏关键接口请求已结束
  4. 页面未发生路由切换、滚动打断、卸载

你在简历里提到，性能预警后期出现了“告警疲劳”，甚至后台只能展示 2 个小时前的数据，后来通过“告警抑制”和“降噪”机制解决了。 ​

你详细讲一下：

  • 当时这个问题是怎么暴露出来的？
  • 到底是发送链路扛不住、存储查询扛不住，还是接收告警的人已经麻木了？
  • 你们的“告警抑制”和“降噪”具体是怎么设计的？
  • 怎么保证降噪之后不会漏掉真正高优先级的问题？ 你尽量按“问题现象 -> 原因分析 -> 机制设计 -> 风险控制”这个顺序回答。

当时线上出现某个问题触发邮件预警，由于是一个大范围的持续性问题，所以导致预警邮件越来越多，写数据（产生告警日志）的速度远大于读数据（展示、处理告警）的速度，且后台处理逻辑存在瓶颈，所以导致数据看不全。

针对这个问题，我们采用了以下方法：

  1. 引入“告警抑制”和“降噪”机制，让系统报警更加智能。第一个是状态翻转机制（Flapping），之前每次检测到性能超标就发邮件，问题恢复又发邮件，反复波动，现在改成只有当问题持续一定时间（例如持续5分钟）或连续发生N次探测失败后，才触发警报。问题恢复后，按告警级别设置不同重发窗口。（比如P2/P3级别的日志24 小时内不再重复发送相同原因邮件）第二个是沉默周期/告警聚合。之前是某个 JS 错误短时间爆发，会发多封邮件，现在改为将同一时间段内（如30分钟内）的同类错误合并为一封邮件，标题注明“有N个接口发生超时”，而非逐一发送。
  2. 实施告警分级与分渠道处理，只在前端展示紧急的实时数据，历史数据通过“查询”功能获取。
  • P0/P1（紧急）： 短信/电话，立即处理。
  • P2/P3（警告）： 邮件，但无需立即展示在前端实时看板，可归档。
  • P4（信息）： 仅入库，供排查问题时查询。
  1. 冷热数据分离
  • 分表/分区： 按天甚至按小时分区。查询2小时内的数据，只需要扫描最近的分区，速度会很快。
  • 数据归档： 将超过24小时或48小时的预警数据，自动转移到归档表或归档数据库（甚至是大数据平台）。前端默认只查询热数据，需要看历史数据时，需要加时间筛选并查询慢一些的归档库。
  1. 限流与熔断：为了在极端情况下，需要保护监控系统自身。引入丢弃策略，即如果同一时刻产生大量重复预警（比如机房断电，上万台机器同时报失联），系统应该能够识别并丢弃非核心的告警，只发送一条“大面积服务器失联”的汇总通知。

通过引入这四个策略，有效解决了这个问题。

“这个问题本质上不是单点故障，而是告警系统在高噪声场景下出现了级联拥塞。 一方面，持续性异常导致大量重复告警被不断生成；另一方面，告警计算、入库、查询和通知链路都被高频重复事件挤占，最终实时看板延迟、通知泛滥、处理人也逐渐麻木。 所以我们的治理思路不是单纯减少告警，而是把‘重复、低价值、短时抖动’的噪声从系统里提前拦截掉，只把持续、扩散、影响大的问题升级出来。”

“相同原因”的判定是怎么做的？也就是你们的告警指纹怎么设计？

比如前端场景里，一个 JS 错误、一类接口异常、一个白屏告警，你怎么判断它们是同一类问题，而不是不同问题？ 我想听你讲：

  1. 错误指纹包含哪些字段
  2. 哪些字段不能直接参与指纹
  3. 怎么避免把本来不同的问题错误聚合到一起
  4. 又怎么避免同一个问题因为环境差异被拆得太散

“我们不会简单拿原始日志全量字段去做指纹，而是先按告警类型设计不同的归一化规则。 对于 JS 错误，核心会取错误类型、归一化后的 message、源码位置、页面路由、发布版本等字段生成指纹。这里 message 会先做清洗，比如把动态数字、用户 ID、订单号这类变量替换掉，避免同一个错误因为动态内容被拆散。行号也不会单独使用，通常要结合 SourceMap 还原后的源码位置和版本号一起看。 对于接口异常，核心会看接口模板路径、请求方法、状态码、业务错误码、调用页面和版本，而不是直接把完整请求参数放进去。参数里很多都是动态值，直接参与指纹会导致同一个接口问题被拆成无数条。通常只会提取少量有分类意义的参数标签。 对于白屏，我们一般不把它直接和其他告警完全合并，因为白屏更像结果态。它会独立告警，但会在分析页里关联同时间窗口内的 JS 错误、接口失败、资源加载异常，帮助定位根因。 为了避免过聚合，我们会保留页面、版本、环境这类关键维度；为了避免过拆分，会对动态字段做归一化处理，并对相似错误做二次聚类。””

SourceMap 是在什么时机上传的？ ​

怎么保证 前端静态资源版本、SourceMap、错误日志 三者能准确对应？ 如果存在灰度发布，用户 A 和用户 B 访问到的不是同一批资源，你怎么保证还原到正确代码？ SourceMap 有源码泄漏风险，你们怎么控制安全性？ 你尽量从“发布链路设计”来讲，这题很适合体现工程化能力。

AI回答： “SourceMap 的关键不是上传，而是建立发布资产和运行时错误之间的精确映射关系。 我们的做法是在 CI/CD 发布阶段生成构建唯一标识，比如 releaseId 或 commit hash，然后把本次构建产物的 JS 文件、对应 SourceMap、版本号、构建时间、环境信息一起上传到内部 SourceMap 平台。 业务页面加载 SDK 时，会把当前运行资源的 releaseId 或构建版本注入到全局环境里，错误上报时带上这个版本标识、报错文件 URL、行列号。 服务端拿到这些信息后，先根据 releaseId + file 找到准确的 SourceMap，再做反解，还原到源码文件、源码行列和函数栈。 如果是灰度发布，我们不会只看业务版本号，而是看用户实际命中的构建批次或资源指纹，这样同一时间不同用户访问不同 bundle 时，也能定位到正确代码。”

安全性部分你还应该补：

“SourceMap 不对公网暴露，只上传到内网平台或私有存储；生产 CDN 不部署 .map 文件；解析服务走权限控制，只允许研发、测试或有权限的人员查看；必要时还会对敏感源码做额外脱敏或限制下载。”

提交态保护 指的是：用户触发提交后，在请求结果返回之前，前端通过状态控制避免重复提交、状态错乱或误操作，把这段“提交中的不稳定窗口”保护起来。提交态保护本质上是在用户发起提交到服务端返回结果之间，前端对关键交互做状态收敛，避免用户因为弱网、卡顿或重复点击造成重复请求和状态错乱。比如按钮置灰、展示提交中状态、限制重复操作、必要时拦截页面离开，这些都属于提交态保护。”“前端的提交态保护解决的是用户侧重复操作和状态错乱问题，重试解决的是瞬时失败恢复问题，后端幂等解决的是重复请求副作用问题，这三者是配套的。

如果面试官追问“幂等是什么”，你就答：

“幂等就是同一个做题动作即使因为超时、重试、重复点击发起多次请求，后端也只按一次有效操作处理，不会重复写入或造成状态错乱。”

如果面试官追问“为什么这组数据可信”，你就答：

“因为统计口径前后一致，都是按同一类会话、同一套异常定义和同一时间窗口去比，而且数据来自监控平台持续观测，不是一次性人工抽样。”

“超时策略、可恢复错误重试、提交态保护、失败兜底” ​

“针对网络抖动场景下做题流程稳定性问题，我们不是只靠单一优化，而是做了四层治理。

  • 第一层是超时策略，对关键链路接口按业务重要性和历史耗时分布设置更合理的超时阈值，避免因为短时波动过早失败。
  • 第二层是可恢复错误重试，只对超时、网络中断、短时服务不可用这类有恢复可能的错误做有限次数重试，而不是无脑重试。
  • 第三层是提交态保护，在用户发起关键操作后，通过按钮禁用、状态锁定和离开拦截等方式避免重复提交和状态错乱。
  • 第四层是失败兜底与后端幂等，前端在多次失败后给出明确反馈或补偿路径，后端保证同一请求重复到达时不会产生副作用。 通过这几层配合，最终把网络抖动场景下的做题流程异常率从 5% 降到 0.3%。”

INP 和长任务监控是什么关系？ ​

如果监控平台发现某个页面 INP 很差，你会怎么一步步定位问题？

我想听你回答：

INP 本质上衡量的是什么 它和长任务、主线程阻塞之间的关系 如果一个页面 INP 差，你会从哪些维度继续下钻 最后可能落到哪些前端问题上，比如事件处理、渲染、第三方脚本、数据量过大等

INP 指的是页面首次交互（点击链接、按钮等）到浏览器实际响应的延迟，它衡量的是用户交互的流畅度，如果页面中存在长任务导致主线程阻塞，那么就会导致 INP 值增加。如果一个页面 INP 差，我会优化用 PerformanceObserver 监控是否有长任务，如果没有长任务，则需要排查业务代码是否有性能问题，比如某些点击事件需要异步执行后才给用户反馈，又没有给 loading 提示，导致用户体验很差。

“INP 反映的是用户交互到下一次可见更新之间的整体响应质量，它不只看第一次交互，而是关注页面生命周期中较差的那批交互表现。 它通常和主线程阻塞高度相关，因为用户输入、事件处理、样式计算、布局、绘制都要竞争主线程；一旦有长任务，交互就会排队，INP 就容易变差。 如果发现某个页面 INP 很差，我会先按交互链路拆解： 第一，先看是不是输入延迟高，也就是交互发生时主线程是否正被长任务占满； 第二，看事件处理本身是否做了重计算，比如同步循环、复杂 JSON 处理、大量状态更新； 第三，看事件触发后是否带来了重渲染、频繁布局计算或大面积 DOM 更新。 继续下钻时，我会结合 PerformanceObserver 看 long task，结合用户行为埋点看是哪一类交互触发问题，再结合性能面板或自定义埋点去定位具体组件、函数和第三方脚本。”

“如果确认是点击一个节点导致 500 条目录树全量重渲染，我会先把问题拆成三层。 第一是数据结构层，检查是不是把整棵树放在一个大响应式对象里，每次展开都返回新树或深拷贝，导致所有节点依赖失效。如果是这样，我会考虑把展开态从树数据中拆出来，单独按节点 id 维护，或者把树做扁平化索引，只更新当前节点相关分支。 第二是组件层，检查是不是父组件持有整棵树状态，导致一次展开把所有子节点 props 都重新传了一遍。这里我会把树节点拆成更细粒度的节点组件，尽量让展开只影响当前分支，同时保证 key 稳定、更新边界清晰。 第三是渲染层，如果目录很长，我不会一次把所有层级节点都挂载出来，而是只在节点展开时按需渲染子树；必要时对超长列表做虚拟化，或者把非关键节点分片渲染，降低单次主线程压力。 优化后我会结合性能面板和监控数据验证，比如看交互时长、长任务、节点渲染数量和 INP 是否明显下降。””

如果这是一个 Vue3 项目，展开目录时整棵树全量更新，你会优先检查哪些响应式使用方式可能导致这个问题？ ​

“在 Vue3 里，如果树展开导致整棵树更新，我会优先检查几类响应式用法。 第一，看是不是每次展开都对整棵树做了深拷贝或重新赋值，这会让所有节点引用变化，导致子组件全量更新。 第二，看展开态是不是直接挂在树节点对象上，并且整棵树被 reactive 深代理了，这样一个节点状态变化可能会把整条依赖链都激活。 第三，看父组件在 render 或 computed 里是不是每次都生成新的数组、对象或过滤后的树结构，导致递归节点组件收到的新 props 引用全部变化。 第四，看 key 是否稳定、是否存在 v-if 和 v-for 组合使用导致节点频繁销毁重建。 如果确认是这些问题，我会优先把展开态独立出去、保证树节点引用稳定、减少整树级别的新对象生成，必要时用 shallowRef 或更细粒度的数据结构来收窄更新范围。””

设计异常上报体系，能够捕获前端 JavaScript 错误、资源加载失败、API 请求异常等，还支持自定义错误上报。” ​

  • JavaScript 运行时错误（包括语法错误、类型错误、引用错误等）和资源加载失败，通过 window.onerror 事件捕获。

“这四类异常我们会分开采集。 第一，JS 运行时错误主要通过 window.onerror 捕获，可以拿到 message、source、line、column 和 error stack。 第二，Promise 未捕获错误通过 window.addEventListener('unhandledrejection', ...) 捕获，拿到 rejection reason，再做统一格式化。 第三，资源加载失败一般通过 window.addEventListener('error', handler, true) 在捕获阶段监听，因为图片、脚本、样式这类资源错误不会像普通运行时错误那样稳定冒泡；再根据事件 target 判断是 img、script、link 等资源，并提取资源 URL。 第四，接口异常一般在前端对 XMLHttpRequest、fetch 或请求库做统一拦截，采集请求地址、方法、耗时、状态码、业务码、错误信息等；必要时再结合后端日志做关联分析，但前端侧本身要先能感知请求失败和超时。”

“我会分层做统一采集，而不是只依赖业务方走同一个 request 封装。 对于 Axios，优先通过请求/响应拦截器采集 URL、方法、耗时、状态码、业务码和错误信息。 对于 fetch，通过重写 window.fetch 做一层包装，在发起前记录开始时间，在返回或抛错时补齐结果和耗时。 对于 XMLHttpRequest，通过代理 open、send 和相关事件，在请求完成、失败、超时、abort 时统一上报。 这样不管业务是直接用原生请求、Axios，还是历史代码混用，都能尽量覆盖。 如果团队里已经有统一 request 层，我会优先复用它补充业务字段；但 SDK 层仍然保留底层兜底采集能力，避免漏数。””

如果你要劫持 fetch，你会怎么保证： ​

不影响原有返回值和调用方式 能拿到请求耗时和状态码 如果请求失败，也能正常把错误继续抛给业务方

“我不会改变 fetch 的调用方式，而是先缓存原始 window.fetch，再用一个包装函数覆盖它。包装函数内部仍然调用原生 fetch，所以对业务侧来说入参和返回值保持不变，仍然拿到一个 Promise<Response>。 调用前先记录开始时间和请求信息，调用成功后读取 response.status、ok 和耗时做上报；如果 fetch 本身 reject，比如网络错误，我会在 catch 里记录异常、耗时和请求信息，然后把原错误继续 throw 出去，保证业务原有的错误处理逻辑不受影响。 核心原则是‘只旁路采集，不改变业务语义’。”

你提到前端 SDK 做到了压缩后 < 30KB，而且采用插件化架构支持错误、性能、行为埋点按需加载。 ​

那你详细讲一下： 这个插件化架构是怎么设计的？ 插件和核心 SDK 之间通过什么方式通信？ 如何保证某个插件出错不会影响业务页面和其他插件？ 按需加载是构建时做的，还是运行时做的？你们是怎么权衡的？

“我们把 SDK 分成核心层和插件层。核心层负责公共能力，比如配置管理、事件总线、上下文采集、批量上报、生命周期管理和异常隔离；插件层只关心各自的采集逻辑，比如 JS 错误、性能指标、行为埋点、接口异常等。 插件注册后会拿到统一的上下文对象和 SDK 提供的能力，比如 report、subscribe、getConfig 等，然后通过事件总线监听诸如 init、routeChange、errorCaptured、performanceReady 这类事件。 为了保证隔离性，插件钩子执行时都会被统一包裹，单个插件抛错只会上报内部错误，不会阻断主流程；必要时还可以对异常插件做降级或熔断。 按需加载上，我们更偏向运行时按配置启用插件，这样接入方可以按业务场景选择能力，减少默认体积；但对最关键的采集，比如全局错误和基础性能入口，会尽量内置或优先初始化，避免因为插件延迟加载而丢失关键数据。””

如果你们把性能监控插件做成“运行时按需加载”，那像首屏性能、初始化阶段 JS 错误这种非常早期的数据，不就可能在插件加载前丢了吗？ ​

你们会怎么处理这个问题？

“这里我会更准确地表述成‘按需启用’，而不是把关键插件真正延迟到很晚再加载。 因为像全局错误监听、首屏性能采集这类能力时机非常早，如果插件晚于 SDK 初始化再异步加载，确实有丢数风险。 所以我们的做法是：在 SDK 初始化阶段就完成核心插件注册，把关键监听入口提前挂上；所谓按需，更多是指接入方可以通过配置决定启用哪些插件能力，而不是等页面运行很久之后再去动态加载。 对于非首屏、非关键路径的能力，比如部分行为埋点或增强分析插件，才可以更灵活地延后初始化。””

FMP 这个指标业界现在争议挺大，很多团队已经不把它当核心指标了。 ​

你们为什么还保留它？实际是怎么采集的？

FMP 对于每个页面都会不一样，这个是根据业务需要决定的，比如我们当时有个抽奖页面，我们就需要统计抽奖转盘的 FMP，而不是页面加载完成的 FMP，作为衡量当前页面性能的重要指标。这个针对特定 DOM 元素的 FMP，在采集时需要传递外层容器元素的 ID 或者类名，内部 sdk 会采集该 DOM 元素的渲染时间。
