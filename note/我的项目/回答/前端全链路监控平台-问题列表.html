速记版

为什么自研而不是用现成监控方案？ 现成方案能覆盖通用错误和基础性能，但很难满足我们对秒开率、白屏、业务关键区域渲染、行为轨迹、SourceMap、内部告警和多业务统一分析的深度定制需求；再加上日志量大、长期成本高，所以最终选择自研。

秒开率是什么？ 秒开率是首屏在 1s 内达到可见可用状态的用户占比。它比单纯看 FCP/LCP 更贴近用户感知，因为浏览器有绘制不代表用户已经看到真实内容。

首屏检测算法怎么做？ 基于 MutationObserver 持续监听首屏区域 DOM 变化，过滤骨架屏和无效节点，结合元素可见性、关键资源加载和页面稳定窗口，判断首屏何时进入“真实可感知且基本稳定”的状态。

为什么 DOMContentLoaded 不能直接当 SSR 首屏时间？ 它只表示初始 HTML 解析完成，不代表关键内容已显示、图片已加载、接口数据已回填，也不代表 hydration 完成，所以只能作为参考时间点。

FMP 为什么还有价值？ 标准 FMP 有争议，因为缺少统一标准。我们保留的是“业务关键区域渲染时间”这种业务化 FMP，用来衡量转盘、价格区、核心购买区等关键区域什么时候真正可见，作为 Web Vitals 的补充。

如何避免业务化 FMP 误判？ 骨架屏不过早计入；关键图片/视频要结合资源加载完成；异步分批渲染要看稳定窗口；必要时结合业务数据完成标记一起判断。

JS 错误、Promise 错误、资源错误、接口异常怎么采集？ JS 运行时错误通过 window.onerror；Promise 未捕获错误通过 unhandledrejection；资源加载失败通过捕获阶段监听 error 事件并识别 event.target；接口异常通过劫持 fetch/XMLHttpRequest 和 Axios 拦截器统一采集。

为什么 fetch 劫持不能影响业务？ 核心原则是“只旁路采集，不改变业务语义”。缓存原始 fetch，包装后继续返回原始 Promise<Response>，成功时记录耗时和状态码，失败时记录异常后继续把错误抛给业务方。

为什么选 ClickHouse 而不是 Elasticsearch？ 我们的核心查询是按时间、页面、版本、设备、错误类型做聚合分析和趋势统计，不是全文检索。ClickHouse 更适合高吞吐写入、多维聚合、分位数统计和低成本存储。

为什么性能数据适合采样，错误数据不适合随意采样？ 性能数据量大，目标是看整体趋势和分布，合理采样仍有统计意义；错误数据更偏发现问题和根因定位，低频高危问题一旦采样掉就可能直接漏报。

动态采样怎么做？ 根据系统压力和业务优先级共同决定。压力接近阈值时降低通用性能数据采样率，但核心业务流程、发版观察期、重点用户分层、新版本系统和特定机型保持高采样率。

告警疲劳怎么解决？ 用告警抑制、聚合、分级、冷热数据分离和限流熔断。核心思想是提前拦截重复、低价值、短时抖动的噪声，只把持续、扩散、影响大的问题升级出来。

SourceMap 怎么做准确映射？ 发布时建立 releaseId -> assets -> sourcemap -> commit 映射；错误上报带 releaseId + file url + line + column；服务端先匹配具体 chunk，再用 .map 反解源码位置，最后结合 commit 生成可读错误详情和仓库跳转链接。

releaseId 是什么？ 是一次前端发布产物的唯一标识，通常由 appName + version + commitSha/构建号 组成，用来把线上运行代码、SourceMap 和错误日志准确关联起来。

行为轨迹怎么帮助排障？ 通过 sessionId/pageViewId/requestId 把页面访问、点击、接口请求、错误和性能异常串成时间线。先判断点击是否发生，再看请求是否发出、是否返回、返回后是否报错，从而区分是交互没触发、接口异常、还是前端处理链路出错。

INP 是什么？ INP 反映的是用户交互到下一次可见更新之间的整体响应质量，不是首次交互时间。它和主线程阻塞、长任务、事件处理耗时、重渲染密切相关。

如果 INP 很差，怎么定位？ 先看是否有长任务占住主线程，再看事件处理本身有没有同步重计算，最后看交互后是否触发大范围渲染、布局计算或第三方脚本抢占主线程。

页面体验评级模型为什么有必要？ 单个指标只能反映体验的一部分。我们把加载体验、交互体验和稳定性体验组合成评级模型，输出优良中差等级和主要短板，让业务能排优先级，研发能快速知道先优化哪里。

课程详情页和后台管理页的权重怎么不同？ 课程详情页更偏转化，加载体验和稳定性权重更高；后台管理页更偏高频操作，交互体验和稳定性权重更高。原则是围绕页面核心业务目标分配权重。

监控平台不只是错误列表，核心页面有哪些？ 总览页负责发现问题，错误分析页负责看影响范围和版本归因，性能分析页负责拆解体验瓶颈，会话回溯页负责还原用户现场，形成从发现到定位的排障闭环。

做题流程优化这条怎么讲更稳？ 不要说“提交接口只有一个却靠后端合并接口解决”，更稳的说法是：监控平台定位到网络抖动场景下做题流程相关接口链路稳定性问题，推动前端优化超时策略、可恢复错误重试、提交态保护和失败兜底，并协同后端完善接口幂等，最终将相关场景下流程异常率从 5% 降至 0.3%。

幂等是什么意思？ 同一个做题动作即使因为超时、重试、重复点击发起多次请求，后端也只按一次有效操作处理，不会重复写入和产生副作用。

提交态保护是什么？ 用户发起关键提交后，通过按钮禁用、状态锁定、离开拦截、明确 loading 提示等方式，避免重复提交、状态错乱和流程中断。

可恢复错误重试是什么？ 只对网络超时、短时抖动、临时服务不可用这类“重试后有机会恢复、且不会带来错误副作用”的错误做有限重试，不对参数错误、业务失败等无意义错误重试。

如果重做这个平台，会改什么？ 更早统一事件模型、公共上下文和错误指纹规则；更清晰地区分平台通用能力和业务定制能力，把业务化指标更多做成插件或配置，而不是直接堆进平台核心。

项目介绍 ​

“我做过一个前端全链路监控平台项目，这个项目的背景是我们当时线上问题比较黑盒，很多问题都要依赖用户截图、录屏或者口述复现步骤，定位效率很低，而且业务核心页面越来越多，单靠人工排查很难支撑。

所以我主导设计并落地了一套公司内部的前端监控 SDK 和可视化分析平台。这个平台不是只做单点报错监控，而是把前端侧从用户访问、页面渲染、用户行为、接口请求、异常上报到告警分析和问题定位这一整条链路打通了。我们最终覆盖了公司主站和后台管理系统 10 多个核心业务，日均处理百万级日志数据。

在这个项目里，我主要做了几块事情。第一块是核心 SDK 设计，我从 0 到 1 设计了一个轻量级 SDK，压缩后控制在 30KB 以内，采用插件化架构，把错误监控、性能监控、行为埋点这些能力解耦，尽量降低对业务代码的侵入。第二块是性能监控，我基于 Performance API 和 Web Vitals 建立了页面性能基线，同时结合业务场景补充了秒开率、业务关键区域渲染时间、卡顿这类更贴近用户感知的指标。第三块是异常监控，统一采集了 JavaScript 运行时错误、资源加载失败、接口异常和自定义业务异常，并结合 SourceMap 实现了错误源码定位。第四块是平台分析能力，基于 ClickHouse 做日志存储和聚合分析，设计了总览、错误分析、性能分析和用户行为路径等核心页面，帮助研发从发现问题快速走到定位根因。

这个项目里我觉得比较有挑战的点有三个。一个是首屏和白屏难量化，我们后来基于 MutationObserver 设计了一套首屏检测算法，结合骨架屏过滤、关键资源加载和页面稳定窗口去判断首屏完成时机。第二个是告警疲劳，随着监控量起来以后，重复告警很多，我们又补了告警抑制、聚合、分级、冷热数据分离和限流熔断机制。第三个是海量数据下的采样和分析，我们针对性能数据做了动态采样策略，在保证统计代表性的同时控制存储和计算成本。

最终这个平台覆盖了 95% 以上的核心页面，把故障发现时效从过去依赖用户反馈的小时级缩短到系统告警的分钟级；结合 SourceMap 后，线上 JS 异常排查从依赖用户复现缩短到可以直接定位源码文件和代码行；另外我们也基于监控平台持续推动业务优化，比如定位到网络抖动场景下做题流程链路的稳定性问题，推动前后端协同优化后，把相关场景下的流程异常率从 5% 降到了 0.3%。”
