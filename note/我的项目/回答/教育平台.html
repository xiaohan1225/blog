3 分钟项目介绍版 ​

“BT 教育平台是一个覆盖 PC 官网、后台、H5、Hybrid 和小程序的综合教育平台，核心业务包括课程、听课、题库、支付、营销和社区等。我在这个项目里担任前端负责人角色，主要负责前端架构设计、核心技术方案落地和工程效能建设。

这段经历里我重点做了四件事。

  • 第一是官网 SSR 性能优化。我们先通过监控和服务端链路拆解，把首屏关键路径拆成 TTFB、接口聚合、SSR render、HTML 返回、首屏渲染和 hydrate 几段，定位到核心瓶颈在接口聚合耗时、实时 SSR 重复计算和首字节返回偏晚。后面我们做了 CDN、Redis、Node 内存三层缓存，结合流式渲染和 ISR，既减少了实时渲染成本，也缩短了用户白屏时间。

  • 第二是微前端改造。当时有 10 多个历史项目并存，技术栈包括 Vue2、Vue3、React 和原生 JS，公共能力分散、系统割裂、协同成本都很高，所以我们用 qiankun 做统一基座，把多个历史系统接到同一入口下，同时保留独立开发和渐进迁移能力。这个过程中重点解决了路由冲突、样式污染、应用通信和子应用卸载不干净的问题。

  • 第三是在线做题场景下的高可靠同步。因为弱网、多端切换、意外关闭会导致答题记录丢失和数据冲突，传统的本地缓存加重试只能提升上传成功率，解决不了并发覆盖问题。所以我们把方案拆成两层：事件溯源负责过程可追踪、可恢复，CRDT 负责主观题文本和草稿区这类可合并数据的并发合并，并配合 snapshot、update 日志和 checkpoint 做增量同步和断点恢复。

  • 第四是AI提效落地。搭建覆盖需求拆解、编码实现、测试验证到 Code Review 的 AI 辅助工作流，显著提升了团队开发效率。

整体上，这段项目最能体现我的，是面向复杂业务系统做架构抽象、方案取舍和落地治理的能力。

重复取数：在 多台 SSR 服务实例 部署的情况下，如果没有共享缓存，同一个页面请求命中到不同机器时，每台机器都会各自去请求后端接口拿同一份数据，这就叫重复取数。

“当时先不是我们主动发现的，而是用户侧先有反馈，主要是首页、课程详情页这类高流量页面在高峰期访问变慢。我们没有直接下结论，而是先把首屏链路拆开看，分别看 TTFB、服务端接口聚合耗时、SSR render 耗时、HTML 返回耗时、浏览器首屏渲染和 hydrate 耗时。

所以后面的优化思路也是顺着这个链路来的：接口层做缓存和字段收敛，SSR 层做多级缓存和 ISR 减少实时计算，返回链路上用流式渲染让 HTML 更早到浏览器，优先把首屏内容吐出去。”

“我们当时还会看高峰期和非高峰期的数据对比，发现高峰期 TTFB 抬升最明显，进一步拆解后主要抬升集中在接口聚合和 SSR render 两段，这也说明单纯做前端资源优化解决不了核心问题。”

回答版本： “当时最早是用户反馈官网有些页面打开很慢，尤其是首页、课程详情页这类高流量页面。我们没有直接猜原因，而是先把首屏链路拆开看，分别去看 TTFB、服务端接口聚合耗时、SSR render 耗时、HTML 返回时间、以及浏览器首屏渲染和 hydrate 耗时。排查下来发现核心瓶颈主要有三块：第一是服务端需要聚合多个接口，部分接口响应不稳定，导致数据准备阶段变长；第二是热门页面很多请求都走实时 SSR，重复取数和重复渲染比较多，服务端 render 压力大；第三是原来要等完整 HTML 生成后再整体返回，首字节出来比较晚，所以用户感知上白屏时间会比较长。基于这个结论，后面我们分别做了接口数据缓存和字段收敛、SSR 多级缓存和 ISR、以及流式渲染优化，整体把首屏体验拉了上来。”

三层缓存分别是什么，各自缓存什么，怎么失效？ ​

  1. CDN缓存：配合 ISR 实现边缘缓存，缓存静态资源和预渲染页面。页面内容更新或者revalidate时间到了，会更新缓存。
  2. 内存缓存：缓存热点数据（高频访问、相对稳定、且所有用户共享的数据，比如导航菜单/分类树、静态模板片段、热门榜单/推荐列表），减少数据库访问，配合 LRU 算法实现缓存淘汰，并设置最大缓存大小。缓存服务重启失效，LRU自动淘汰。
  3. redis缓存：缓存动态数据，如用户会话、登录状态等。跨服务器共享，持久化。数据更新后会主动删除redis让缓存失效。

“我们当时做的是三层缓存。第一层是 CDN 边缘缓存，主要缓存静态资源和适合缓存的 HTML 页面，目标是尽量减少回源，让用户就近拿到内容。第二层是 Redis 分布式缓存，主要缓存 SSR 过程中聚合后的接口数据，以及部分可复用的页面 HTML，这一层是给多实例共享的，避免每台机器都重复取数、重复渲染。第三层是 Node 进程内缓存，主要放热点接口结果、模板片段和少量热点页面，追求极致低延迟，用来扛瞬时高并发。

失效策略上我们主要是 TTL 加主动失效结合。像活动规则页这类低频变更页面，会配合 ISR 或定时过期；像课程价格、库存这类实时性高的数据，更新后会主动删 Redis，并同步让本地内存缓存失效。个性化内容我们不会直接缓存完整 HTML，而是把公共部分缓存下来，用户态数据实时渲染，避免缓存脏数据。”

策略 全称 淘汰依据 类比 适用场景 LFU Least Frequently Used 访问频率最低 图书馆借阅次数最少 热点数据识别 LRU Least Recently Used 访问时间最久远 最近没翻过的书 时间局部性强的场景 FIFO First In First Out 进入缓存的时间最早 先来的先走 数据有效期固定

缓存踩坑：

  1. 有人共用同一份缓存出问题
  2. 不同版本有时候不能共用同一份缓存，需要配合脚本清除redis缓存

ISR ​

  • 预生成热门页面
  • CDN集成，实现边缘缓存

ISR的"Stale-While-Revalidate" ISR实现的是业界标准的SWR模式：

  • 用户A（第61秒访问）：立即看到旧页面（瞬间响应）
  • 后台触发：框架静默重新生成新页面
  • 用户B（下一秒访问）：看到新页面

这种"先返回旧数据，后台更新缓存"的模式，让用户永远不需要等待。

自己写缓存的常见困境 同步阻塞：很多简单实现是"缓存过期后，请求必须等待新数据生成"

缓存击穿：当缓存刚好过期，100个并发请求同时打到数据库

需要自己实现锁机制来防止缓存击穿

流式渲染和 ISR 在你们项目里分别解决什么问题？为什么两个都要用？ ​

“这两个解决的不是同一类问题。流式渲染主要解决的是返回时机问题，原来要等完整 HTML 全部生成后再一次性返回，现在可以边生成边输出，让浏览器更早拿到首字节和首屏内容，所以用户感知上的白屏时间会更短。ISR 主要解决的是实时 SSR 成本高的问题，尤其是一些访问量大但内容更新没那么频繁的页面，没必要每次请求都重新取数和渲染，可以按周期或在内容更新后重新生成，这样能明显减少服务端计算压力。

所以两个都要用，因为它们作用的阶段不一样。ISR 是减少服务器每次现算的压力，流式渲染是让已经要返回的内容更早到达浏览器。前者解决‘算得太多’，后者解决‘返回太晚’，合起来才能同时优化服务端性能和用户首屏体验。”

首屏时间从 2.1s 到 0.5s，你怎么证明这个收益是你的方案带来的，而不是别的因素？ ​

“这个收益我们不是只看一次前后对比，而是尽量做了统一口径验证。首先，2.1 秒和 0.5 秒都是同一套性能监控口径下统计的同类页面数据，不是随便拿两个时间点对比。其次，SSR 是灰度上线的，我们在灰度阶段对比了新旧链路的 TTFB、首屏时间和服务端 render 耗时，确认提升主要集中在 SSR 相关链路上。再加上缓存命中率提升、实时渲染请求占比下降，这些过程指标也能说明优化确实生效了。因为同期也有接口字段精简和资源优化，所以我们不会说 0.5 秒完全是某一个点带来的，而是这套 SSR 架构优化整体产生的结果。”

“我们做过灰度对照，不是全量 A/B 实验。灰度阶段同一类页面下，新链路相对于旧链路在 TTFB 和首屏时间上都有明显下降，等稳定后再全量上线。”

最后在效果验证上，我们不是只看一次简单的前后对比，而是尽量用统一口径去验证。首先，优化前的 2.1s 和优化后的 0.5s，都是基于同一套性能监控口径统计的同类页面数据。其次，SSR 是灰度上线的，我们在灰度阶段对比了新旧链路的 TTFB、首屏时间和服务端 render 耗时，确认下降主要集中在 SSR 相关链路上。同时像缓存命中率提升、实时渲染请求占比下降，这些过程指标也能证明优化确实生效了。因为同期可能也有接口字段精简、资源加载优化等动作，所以我们一般不会说 0.5 秒完全是某一个点带来的，而是这套 SSR 架构优化整体带来的结果。”

为什么当时要做微前端改造？原来的“巨石应用 / 多项目并存”具体痛点是什么，为什么不是继续在原架构上修修补补？ ​

“微前端不是为了把一个系统拆得更碎，而是为了处理历史多系统并存、多技术栈共存、发布耦合和统一门户这些结构性问题。它解决的是组织和架构层面的边界问题，不解决代码质量本身，后者还要靠工程规范、组件复用和基座治理去补。”

“当时做微前端改造，核心不是为了追新技术，而是原来的组织方式已经撑不住业务发展了。我们当时有多个历史前端项目并存，技术栈也不统一，有 Vue2、Vue3、React 甚至原生 JS，很多系统是独立开发、独立部署的。这样带来的第一个问题是维护成本高，很多公共能力比如登录、权限、埋点、导航，在不同项目里其实都有各自的实现，一旦协议或者逻辑变更，就要在多个项目里重复修改，而且很容易改得不一致。第二个问题是整体协同成本高，虽然系统表面上是分开的，但很多业务链路是连通的，一个公共能力升级或者一次线上发版，经常要多个项目一起联动回归，风险和沟通成本都很高；第三个问题是用户体验割裂，用户在不同系统之间要反复切换，导航、登录态、菜单风格都不统一。第四个问题是历史系统体量大、耦合深，不适合一次性重构，比如 Vue2 升 Vue3 很难整体切换，只能找一种让新旧系统并存、渐进迁移的方案。

我们也评估过继续在原架构上修补，比如统一组件库、统一规范、逐步升级技术栈，但这些方案只能解决局部问题，解决不了多技术栈长期并存、系统割裂和协同发布成本高这些结构性问题。所以最后才选了微前端，用统一基座把多个历史系统整合起来，同时保留子应用独立开发、独立部署和渐进迁移的能力。当然微前端也不是银弹，它会带来运行时复杂度和治理成本，所以只有在我们这种历史系统多、异构严重、又需要统一入口的场景下，收益才明显大于成本。

原系统存在问题：

原系统不是一个单体项目，而是多个历史项目并存、彼此割裂的状态，在公共能力上，会散落在各个项目里各自维护。公共能力：

  • 登录态校验
  • 权限判断
  • 埋点上报
  • 菜单和导航
  • 请求封装
  • 错误处理
  • 主题样式
  • 上传、导出、支付这类通用流程

“这里的维护成本高，不是单指一个项目代码多，而是多个历史系统并存后，很多本该统一的公共能力其实分散在各个项目里各自维护。比如登录、权限、埋点、导航这些能力，一旦协议或者逻辑变了，就得在多个项目里重复修改，而且因为技术栈和实现方式不一样，很容易改得不一致，最终既增加维护成本，也影响用户体验。”

为什么选 qiankun，而不是 iframe、Module Federation，或者自己做一套？ ​

技术价值观：“在做技术选型时，我通常遵循 ‘匹配度优先’和 ‘成本可控’原则。没有银弹，所有的选择都是在特定约束条件下的最优解。对于当时的场景，选择 qiankun 是综合了业务稳定性、团队熟悉度、生态成熟度以及维护成本后的结果。”

Iframe：确实提供了完美的硬隔离，但它带来的割裂感是业务无法接受的

iframe 优点：

  1. 独立性：每个 iframe 都是一个独立的页面，有自己生命周期和上下文，可以独立运行，独立开发，独立部署，互不干扰。
  2. 实现简单：只需要通过 iframe 标签即可实现，无需复杂的配置和代码。

iframe 缺点：

  • 用户体验断层： 浏览器的刷新、前进后退无法与主应用的路由天然同步，导致用户感觉‘跳到了另一个网站’。弹窗只能在 iframe 内局限，无法居中于整个大屏。
  • 应用通信复杂： 虽然可以用 postMessage，但数据同步往往需要写大量胶水代码，且无法共享一些基础上下文（如用户鉴权信息、UI 主题）。

在这次微前端改造里，样式冲突和路由冲突你们分别是怎么解决的？ ​

样式冲突我们不是只靠 qiankun 本身解决的，而是做了分层治理。qiankun 的运行时隔离能帮我们处理一部分容器范围的问题，但 CSS 冲突本质上还要靠约束和改造。我们当时会要求子应用尽量避免全局样式污染，统一使用命名空间或局部样式方案，比如给应用根节点加前缀作用域；对一些历史项目里的全局 reset、组件库样式覆盖、挂到 body 的弹窗，也做了专项处理，保证它们尽量挂到子应用容器内。这样才能真正避免子应用之间互相覆盖。

路由冲突这块，我们核心做了两件事。第一是统一路由边界，每个子应用都有独立的路由前缀，主应用通过 activeRule 做分发，避免多个应用匹配到同一段路径。第二是统一路由运行规范，子应用在独立运行和接入主应用时会用不同的 base 配置，保证本地开发、单独部署和接入基座后都能正常工作。除此之外，路由监听和销毁也要跟生命周期绑定，避免应用卸载后还残留监听逻辑，导致重复跳转或状态错乱。

  1. 动态样式隔离（sandbox: true 或 {}） 这是 qiankun 最基础的样式管理，更像是一种“样式生命周期管理”。它的逻辑很简单：当子应用激活时，就把它的所有样式通过 <style> 标签插入页面；当子应用卸载时，就把这些标签移除 。这能保证在任何时刻，页面上只有当前活跃子应用的样式，从而避免了子应用之间的样式冲突。但它无法处理主应用和子应用样式共存时可能发生的冲突。

  2. 作用域沙箱（sandbox: { experimentalStyleIsolation: true }） 你可以把它想象成 Vue 里的 scoped 概念的加强版。它的原理是通过一个叫 import-html-entry 的库，在加载子应用时，解析其所有样式（包括内联和外联），并为每一条 CSS 规则的选择器前面，都加上一个类似 div[data-qiankun="your-app-name"] 的属性选择器 。

优点：这样一来，所有样式都被限制在了你为子应用指定的那个容器内，不会泄漏出去。

痛点与解法：正如搜索结果中提到的，这种方案会导致 Ant Design 5.x 等组件库的样式失效 。这是因为 antd 5.x 为了降低样式覆盖成本，默认使用了低权重的 :where 选择器，很容易被子应用容器上的 data 属性选择器覆盖。解决方法是使用 antd 的 StyleProvider 将 hashPriority 设为 high 来提升权重，或者利用 getContainer 将弹窗这类组件强制渲染在带有 data-qiankun 属性的容器内 。

  1. Shadow DOM 沙箱（sandbox: { strictStyleIsolation: true }） 这是最彻底的隔离方案。它为每个子应用创建一个独立的、封闭的 Shadow DOM 作为渲染容器 。在 Shadow DOM 内部，样式和 DOM 与外界完全隔绝，就像一个小型浏览器环境。

优点：隔离效果最好，无论子应用内部如何定义样式，都不会影响主应用。

痛点：这种强隔离也带来了一些挑战。比如，Ant Design 的 Modal 默认通过 Portal 渲染到 document.body，这就“逃”出了 Shadow DOM，导致其无法应用子应用内部的样式 。你需要手动修改这些组件的挂载节点，让它们渲染回 Shadow DOM 内部。

qiankun 的 CSS 方案设计思路是从“软约束”到“硬隔离”，你可以根据项目的实际情况按需选择：

如果项目是全新的，且子应用都遵循统一的命名规范：使用默认的动态样式隔离就足够了，性能好，简单高效。

如果需要隔离主应用和子应用，但改造子应用的成本较高：优先尝试作用域沙箱。它侵入性低，但要注意测试并解决可能出现的第三方库样式丢失问题。

如果对隔离有极端要求，且团队有能力处理框架兼容性问题：可以考虑 Shadow DOM 沙箱，但必须做好技术调研和充分的兼容性测试，并接受它可能带来的维护成本。

通常在实际的大规模应用中，更稳妥的做法是组合使用：“默认隔离” + “子应用自身使用 CSS Modules 等工程化手段”，这样既能保证基础隔离，又能通过规范来规避大部分问题

应用间通信和状态同步你们是怎么做的？为什么不用直接共享全局 store？ ​

应用间的通信一般有两种方案：

  1. actions：主应用通过 qiankun 提供的 API，初始化一个全局状态，并将状态和状态更新的接口通过props传递给子应用，同时可以通过监听状态变化获取新的状态，本质上就是通过 全局状态共享 + 发布订阅模式 进行状态同步。
  2. 全局shared。比如用一个三方状态库 redux，在主应用初始化并传递给子组件。

“我们当时没有把应用间通信做成一个大而全的共享 store，而是按场景分层。像登录态、用户信息、租户信息、主题这类全局且变化频率不高的状态，我们放在主应用，通过 qiankun 的 initGlobalState 和 props 下发给子应用，子应用只订阅自己关心的部分。像页面跳转、刷新菜单、埋点通知这类一次性的业务事件，我们不会放到全局状态里，而是走事件通知或者主应用统一调度。

我们没有直接共享一个全局 store，比如让所有子应用都依赖同一个 Redux，核心原因是这样耦合太重。第一，我们当时技术栈是异构的，有 Vue2、Vue3、React、原生 JS，强行共享同一个 store，会把各个子应用绑死在同一套状态实现上。第二，微前端强调独立开发、独立部署，如果共享全局 store，状态结构一变，多个子应用都可能要一起改，发布解耦就被破坏了。第三，共享 store 容易让状态边界变模糊，谁拥有数据、谁负责更新、谁导致了问题会越来越难排查。 所以我们的原则是：共享状态尽量少，只放真正跨应用的公共上下文；业务状态尽量留在子应用内部，避免把微前端重新做成一个分布式巨石应用。”

你们微前端改造里踩过的最大坑是什么，最后怎么解决的? ​

结合 样式污染 / 路由切换异常 / 子应用卸载不干净 / 公共依赖冲突 / 首屏性能变差 里你最熟的一项来答

“如果让我讲微前端改造里最大的坑，我会讲‘子应用卸载不干净导致的全局副作用残留’。因为一开始我们觉得最难的是接入，但真正上线后发现，很多历史项目其实不是接不进去，而是接进来以后在主应用里反复切换会出问题。

当时我们有 Vue2、Vue3、React 甚至原生 JS 的历史项目，这些项目以前默认都是整页应用，生命周期就是页面加载一次、销毁一次，所以很多代码天然假设自己只会初始化一次。接到 qiankun 之后，子应用会被反复 mount 和 unmount，这时候问题就暴露出来了。比如有的项目在 window、document 上绑定了事件监听但没有释放；有的页面有定时器、轮询、WebSocket、store 订阅没有清理；还有一些弹窗、抽屉、消息提示默认挂在 body 上，应用切走以后 DOM 还留着。线上表现就是用户在几个子应用之间切换几次后，会出现重复请求、埋点重复上报、事件触发多次、内存上涨，甚至页面状态错乱。

这个问题难点在于它不是单个 bug，而是一类问题。而且不同技术栈、不同历史时期的项目，残留方式都不一样，只靠人工排查效率很低。所以我们后面不是单点修，而是把它做成一套治理方案。第一步是梳理所有高频副作用场景，像全局事件、定时器、路由监听、第三方 SDK、WebSocket、挂到 body 的弹层这些，统一形成接入检查清单。第二步是把这些清理动作和子应用生命周期强绑定，要求所有子应用在 unmount 阶段必须完成资源释放；同时我们也封装了一层公共能力，像事件注册、定时器管理、弹层挂载容器这些尽量走统一封装，减少每个项目自己处理。第三步是在联调和验收阶段增加反复挂载/卸载测试，专门观察重复请求、事件次数和内存变化，确保问题能在上线前暴露。

最后这个问题收住以后，应用切换稳定性提升很明显，重复请求和重复埋点这类问题基本被消掉了。对我来说，这个坑也让我更明确一件事：微前端最难的往往不是‘把应用挂上去’，而是怎么把历史项目改造成真正符合生命周期约束的子应用。”

1 分钟口语版

“微前端改造里我觉得最大的坑不是接入，而是子应用卸载不干净。因为很多历史项目以前都是整页应用，只考虑初始化，没认真处理销毁。接到 qiankun 后，子应用会反复 mount 和 unmount，这时候像全局事件、定时器、轮询、WebSocket、store 订阅、还有挂到 body 上的弹窗这些问题就都出来了。用户在几个子应用之间切换几次后，会出现重复请求、埋点重复上报、事件重复触发，甚至内存上涨。后来我们没有按 bug 一个个修，而是把它做成统一治理：先梳理副作用清单，再要求所有子应用把清理逻辑和 unmount 生命周期绑定，同时封装统一的事件、定时器和弹层管理，最后加上反复挂载卸载的专项验收。这个问题解决后，系统稳定性提升很明显。”

  • 这不是单个 bug，而是一类全局副作用治理问题。
  • 解决方式不是人工逐个修，而是规范 + 封装 + 验收一起做。

追问：那你们是怎么把这个治理方案落到 10+ 子应用上的？

“我们当时不是让每个子应用自己去记住这些坑、自己修，而是把这件事做成了一套可落地的治理方案。因为如果只是靠口头约定，10 多个历史项目、多个技术栈，最后一定会反复踩同样的问题。

第二步，我们尽量把治理从‘人工规范’往‘公共封装’迁移。比如事件注册、定时器管理、弹层挂载容器、应用生命周期适配这些能力，我们都做了统一封装，要求子应用优先走公共能力，而不是每个项目自己写一套。这样做的好处是，清理逻辑可以和 unmount 生命周期天然绑定，减少历史项目遗漏的概率。

第四步，我们把它变成可验证的门禁，而不是只靠 Code Review。比如联调和验收阶段，我们会专门做反复 mount/unmount 的切换测试，观察有没有重复请求、重复埋点、事件多次触发、DOM 残留、内存持续上涨这些问题。高频应用还会做灰度观察，结合监控平台按 appName 看错误率、切换稳定性和资源释放情况。

最后在推进方式上，我们也不是一次性全量推，而是先挑几个典型项目做试点，把规范和封装打磨稳定，再批量推广到剩余子应用。这样做下来，接入效率会越来越高，后面的项目基本不是重新踩坑，而是按既有规范接入。对我来说，这件事最重要的经验就是：微前端治理要想在 10+ 应用里真正落地，不能只靠‘大家注意一下’，一定要做成‘规范 + 封装 + 门禁 + 灰度’的体系化方案。”

CRDT ​

CRDT 负责收敛，业务层负责判冲突 "待办清单"或者墓碑机制

同一用户多设备切换，且可能并发编辑 弱网/离线后恢复，并且要自动 merge

在这套同步架构里，update 日志记录的是每次变更，主要用于增量同步和回放；snapshot 是某个时刻的完整状态，主要用于快速恢复；checkpoint 是同步位点，表示双方已经同步到哪一步，用来做断点续传和差异对账。”

你为什么会想到用 事件溯源 + CRDT？原来的答题记录丢失和数据不一致，具体是怎么发生的？为什么普通的“本地缓存 + 接口重试”不够？ ​

你尽量围绕这几个点答：

  • 用户真实场景：弱网、多端切换、意外关闭
  • 原来系统是怎么存和怎么同步的
  • 为什么会丢数据、冲突、覆盖
  • 为什么常规方案解决不彻底

事件溯源是一种数据存储和状态管理的方式，它的核心理念是：不保存对象的当前状态，只保存所有改变状态的事件（Events）。

适合放进 Yjs 的数据：

主观题答案文本 草稿区 批注 光标、选区、presence 状态 题目标记、收藏、折叠状态

“当时答题记录丢失不只是‘没保存上去’，而是原来的数据模型本身在弱网和多端场景下不够稳。我们原来更多是把当前答案状态直接写服务端，客户端再配一个本地缓存和失败重试。这个模型在单端、网络稳定时问题不大，但一旦出现多设备同时作答、弱网重传、离线恢复或者页面异常关闭，就会有两个典型问题：一个是数据来不及上报，导致本地和服务端状态不一致；另一个是多个端对同一道题都有修改时，服务端只能按请求到达顺序或最后一次写入覆盖，容易出现 A 端的修改把 B 端覆盖掉，用户感知上就是‘我明明写过，结果没了’。

所以我们后面没有再只做‘本地缓存 + 重试’，因为它只能提高上报成功率，解决不了并发冲突。我们把方案拆成两层：一层是事件溯源，把作答过程里的关键操作都记录成事件，保证每次修改都有迹可循，也方便断网恢复和状态重建；另一层是对主观题文本、草稿区这类可合并数据引入 CRDT，比如基于 Yjs 做增量同步，这样多端并发编辑时不会简单变成最后写入覆盖，而是尽量做可合并同步。这样才真正把‘数据丢失’和‘并发冲突’两个问题一起解决掉。”

这一题想拿更高分，记住这 3 句

  • 传统方案解决的是上传成功率，不解决并发一致性。
  • 事件溯源负责可追溯、可恢复，CRDT 负责可合并。
  • 不是所有答题数据都用 CRDT，而是按数据类型分治。

你们为什么同时用了“事件溯源”和“CRDT”？这两者分别负责什么，如果只用其中一个行不行？ ​

  • 事件溯源 解决的是“过程可追踪、可恢复、可重放”
  • CRDT 解决的是“多端并发修改时如何合并，避免覆盖”

“这两个方案解决的是不同层面的问题。事件溯源主要解决的是作答过程里的状态流转问题，我们把切题、作答、修改、暂存、提交这些关键操作记录成事件，这样即使在弱网、离线、异常关闭的场景下，也可以基于事件流去恢复本地状态、补发服务端，保证过程可追踪、可恢复。CRDT 主要解决的是并发冲突问题，尤其是主观题文本、笔记区这类会被多端同时编辑的数据，如果只按最后写入覆盖，很容易把另一端的修改顶掉，所以我们用 CRDT 做增量合并，尽量保证 A 和 B 的修改都能保留下来。

如果只用事件溯源，不做 CRDT，那我们只能知道‘发生过哪些修改’，但多端同时改同一份内容时，还是得自己处理冲突，最后很可能还是落回覆盖策略。如果只用 CRDT，不做事件溯源，虽然能解决文本合并，但像切题、暂存、提交、恢复这些业务过程就缺少可追踪和可重建能力，断网恢复和问题排查都会比较弱。所以我们最后是把它们组合起来，一个负责过程可靠性，一个负责并发一致性。”

不是所有答题数据都适合用 CRDT。你们当时是怎么做数据分层的？哪些数据用了 CRDT，哪些没有，为什么？ ​

“我们当时做数据分层，核心判断标准不是数据类型本身，而是这个数据在多端和弱网场景下，是否需要做并发合并。第一类是适合用 CRDT 的数据，比如主观题文本、答题表格、草稿区、答题笔记这类内容，它们可能被多端同时编辑，而且用户通常希望两边修改都尽量保留，这类数据就适合用 CRDT 做增量合并。第二类是不适合用 CRDT、但需要可追踪的业务状态，比如客观题作答、切题、暂存、提交这类操作，我们更适合把它建模成事件，通过事件流和版本控制来保证顺序和恢复能力。第三类是强一致的结果数据，比如最终提交状态、判题结果、得分，这类数据最终以服务端权威状态为准，不在客户端做自由合并。

所以不是所有答题数据都上 CRDT，而是只把真正需要 merge 的内容型数据放进 CRDT；离散状态和关键业务结果，还是用事件模型和服务端权威机制处理，这样复杂度和收益才匹配。”

你们的增量同步协议是怎么设计的？弱网、断网重连、多端切换时，客户端和服务端怎么知道彼此差了哪些数据，而不是每次全量同步？ ​

高分版回答

“我们当时的增量同步协议不是简单地把整份答题数据反复上传，而是把‘同步进度’也纳入协议设计。对于可合并的数据，比如主观题文本和草稿区，我们基于 CRDT 只上传变更后的增量 update，不做整份文档全量同步。每个客户端都会有自己的 clientId/deviceId，同时维护本地的同步进度，比如 seq、lastAck，如果是 Yjs 这类 CRDT 文档，还会维护对应的 state vector。这样客户端在同步时，不是说‘这是我现在的完整答案’，而是说‘这是我相对于上一次确认状态新增的那部分更新’。

服务端这边会按用户、试卷、题目维度保存 update 或事件日志，并做幂等处理。客户端上传增量时会带上自己的身份和当前同步位置，服务端收到后先做去重，再把缺失的 update 合并进服务端文档。如果客户端断网后重连，或者用户从另一个设备切回来，本地会先把未确认的 update 放进待同步队列，然后带着自己的 checkpoint 或 state vector 去和服务端对账。服务端根据这个状态判断双方各自缺了哪些 update，再只回传差异部分，而不是整份全量数据。

这样做的好处是三点：第一，弱网下传输量更小，不会每次都上传整份答题内容；第二，断网重连时可以基于待同步队列和同步位点继续补发，不容易丢；第三，多端切换时可以基于同一个文档的增量更新做并发合并，避免简单按最后写入覆盖。只有在极少数情况下，比如本地状态损坏、服务端日志压缩导致增量无法补齐，我们才会降级走一次快照加后续增量同步。”

1 分钟口语版

“我们做的不是全量同步，而是增量同步。客户端会给每次更新打上自己的 clientId 和同步序号，同时维护 lastAck 或者 state vector。像主观题文本、草稿区这种 CRDT 数据，我们只上传增量 update，不上传整份内容。服务端按题目维度保存这些 update，并做去重和合并。断网重连或者多端切换时，客户端会带着自己的同步位点去和服务端对账，服务端根据这个位点只回缺失的 update，所以不是每次都全量拉整份答题数据。只有本地状态损坏或者增量补不齐时，才会降级走快照。”

你说用的是 Yjs，那服务端到底存的是什么？是存整份文档、存 update 日志，还是两者都存？ ​

“服务端我们是 snapshot + update 日志 两层都存。snapshot 主要用来做快速恢复和降低重放成本，避免每次都从最早的一条 update 开始回放；update 日志 主要用来支持增量同步、并发合并和问题追溯。像客户端断网重连、多端切换时，服务端可以基于 update 日志只下发差异更新，而不是每次返回整份文档。

为了避免 update 日志无限增长，我们会定期做一次压缩，也就是基于当前文档状态生成新的 snapshot，把已经合并进快照的旧 update 做归档或裁剪。触发条件一般会看 update 数量、时间窗口和文档体积。这样恢复时走的是‘最近快照 + 后续增量’这条链路，既保留了增量同步能力，也能把存储和重放成本控制住。”

你们怎么保证 update 日志不会重复应用？也就是幂等和去重是怎么做的？ ​

这块我们分两层看。第一层是 CRDT 层，Yjs 的 update 应用本身就是幂等的，同一个 update 被重复 apply 不会把文档状态搞乱，它内部会基于版本信息和因果关系安全合并。第二层是我们自己的同步协议层，虽然 Yjs 能保证文档级幂等，但为了处理断网重试、服务端确认、日志追踪和断点续传，我们还是会给 update 增加自己的 updateId/seq，并维护 ack/checkpoint。这样服务端既能安全接收重复 update，又能知道哪些已经确认、哪些还需要补发。 所以严格来说，Yjs 解决的是文档状态的幂等合并；协议层还要负责传输去重和同步进度管理。我们真正要求稳定持久化的，更多是 deviceId 或同步位点，而不只是 Yjs 内部的 clientID。

如果服务端和客户端的数据已经出现分叉，你们怎么做冲突恢复和回放？什么时候走增量补齐，什么时候直接回退到 snapshot 重建？ ​

“我们会先基于客户端和服务端的 checkpoint/state vector 做一次对账，先判断分叉程度，再决定恢复策略。 如果是轻度分叉，比如只是某个端短暂离线、双方日志连续，而且 update 还没被压缩，那就直接走增量补齐，把缺失的 update 拉回来交给 Yjs 自动合并，这是最常见也最理想的情况。 如果是中度分叉，比如客户端离线时间比较长、双方都有一段未同步更新，但服务端还保留完整 update 日志，我们会基于同步位点回放缺失增量，必要时按题目维度做顺序校正和幂等去重，再交给 CRDT 合并。 如果是严重分叉，比如日志已经被压缩、版本跨度过大、或者本地状态明显落后，我们就不再强行增量追，而是直接下发最新 snapshot，让客户端基于快照重建，再补后续增量。 如果再极端一点，比如快照校验失败、文档结构损坏、提交状态和答题内容出现矛盾，这时候会走强制恢复和异常标记，由系统告警或人工介入排查。 所以整体原则是：能增量补齐就不要全量重建，只有在日志不完整、状态不可信或者数据损坏时，才回退到 snapshot 重建。””

你们最后说“数据丢失率从 5% 降到 0.03%”，这个指标是怎么定义和统计的？ ​

“这个指标我们不是按投诉数去估算的，而是先统一定义了‘答题记录丢失’：如果用户本地最新一次有效作答状态，在服务端最终恢复状态或提交状态里缺失，且这种差异不是用户主动修改导致的，我们就记为一次丢失。统计口径上，我们一般按发生过作答或编辑行为的题目会话数做分母，而不是简单按请求次数算，这样更能反映真实用户体验。

验证方式上，我们不是只看单次前后对比，而是走统一口径的灰度验证。灰度阶段新旧链路同时跑，对比同类用户、同类题型、同类网络环境下的数据丢失率；同时结合客户端本地状态、服务端恢复结果、最终提交结果三方做交叉校验。上线后也持续观察了一段时间，确认指标稳定下降，才最终得出从 5% 降到 0.03% 这个结论。所以这个结果不是单点观察，而是统一口径下长期验证出来的。”

如果用户在两台设备上同时编辑同一道主观题，最后又在其中一台提交了，服务端怎么裁决“最终答案”？CRDT 合并后的内容一定能直接当提交答案吗？ ​

回答要点：

  • 编辑态数据和提交态数据是不是同一套
  • 最终以客户端还是服务端为准
  • 提交那一刻怎么“冻结版本”
  • 另一台设备后续再同步上来的 update 怎么处理

“我们会把编辑态和提交态分开处理。编辑过程中，多端主观题内容可以用 CRDT 做合并，这样能保证草稿状态最终收敛，不会简单出现 A 覆盖 B。 但提交是一个业务上的‘定版动作’，不能只看 CRDT 最终合并结果。某台设备发起提交时，客户端会带上当前题目的文档状态和同步位点，比如 checkpoint 或 state vector，服务端先把已收到的增量 update 合并成当前权威草稿，再基于这个时点生成一份不可变的 submission snapshot，作为这次提交的最终答案，同时记录 submission version 和提交时间。也就是说，最终提交答案以服务端在提交时刻确认下来的版本为准，而不是以某个客户端本地看到的内容为准。

另一台设备后续再同步时，会拿到最新的题目状态和 submission version。如果产品定义是‘提交即锁定’，那提交后的新 update 不会再直接改动已提交答案，而是提示用户当前答案已在其他设备提交；如果业务允许二次修改，那这些后续 update 只能进入新的草稿版本，用户需要显式再次提交。 所以 CRDT 解决的是编辑态的并发合并问题，提交态还需要业务层去定义版本冻结、状态流转和最终裁决规则，不能把 CRDT 合并结果直接当成最终提交结果。”

你们这个方案里，客户端本地为什么要做“三层存储架构”？分别是什么，为什么不能只用内存或者只用 IndexedDB？ ​

第一层：内存层

  • 实时响应：用户操作的即时反馈
  • 冲突合并：CRDT算法的实时计算
  • UI状态：当前页面展示所需的所有数据

第二层：本地持久化层

  • 离线支持：断网时继续工作
  • 快速启动：二次加载无需从服务器拉取
  • 本地缓存：减少网络请求

第三层：服务端存储层

  • 永久存储：所有数据的最终归宿
  • 多端同步：不同设备间的数据桥梁
  • 数据分析：做题行为分析、统计报表

内存和 IndexedDB分别对应不同场景，内存是为了实时响应用户操作进行乐观更新，保证用户的操作体验，而IndexedDB则是做本地持久化，以应对离线断网等异常情况。

AI回答：

“我们当时做三层存储，不是简单把同一份数据存三遍，而是让每一层承担不同职责。第一层是内存层，主要负责当前会话里的高频读写、CRDT 的实时 merge 和 UI 即时响应，用户输入、切题、撤销这些动作都先在内存里完成，这样交互延迟最低。第二层是本地持久化层，比如 IndexedDB，主要负责离线可用、异常恢复和断点续传。也就是说，即使用户突然断网、刷新页面、应用异常关闭，本地仍然能把最近一次有效状态和待同步 update 保留下来。第三层是服务端存储层，负责跨设备同步、最终提交落库和服务端权威状态维护，也是后续分析和回放的基础。

之所以不能只用一种存储，是因为它们解决的问题不一样。只用内存的话，一旦页面刷新、崩溃或者切设备，本地状态就没了；只用 IndexedDB 的话，高频编辑都直接落盘，性能和交互体验会受影响，CRDT 的实时合并也不适合完全在持久层做；只依赖服务端又扛不住弱网和离线场景，用户体验会很差。所以我们最后是把它设计成内存负责实时、IndexedDB 负责兜底、服务端负责同步和权威状态，这样可靠性和性能才能兼顾。”

“落盘”这个词本来是后端/存储领域里常用的，意思是把数据真正写入持久化介质，不只放在内存里。先在内存里实时更新，保证输入和 UI 非常顺滑，再批量/节流写入 IndexedDB，比如每隔几百毫秒、切题时、页面隐藏时、一定 update 数量后再持久化一次

三层存储不是冗余设计，而是把实时性、可靠性和跨端一致性拆到不同层分别解决。
